### 1.1 未修复的 N+1（需优先处理）

#### 🔴 N+1-P1 — `MicroSpecialtyEnrollmentServiceImpl.java`

**位置**：4 处 for 循环中逐条 repository 查询

| 行号 | 触发场景 | 循环内调用 | 影响估算 |
|------|---------|-----------|---------|
| **L195–L222** | `calcCompletionRate()` — 遍历 `requiredCourses` | `courseEnrollmentRepository.selectOne()` 逐条检查是否已通过 | 30门必修课 × 每次1查询 = **30次额外查询** |
| **L395–L417** | `buildStudentCoursesVO()` — 遍历 `courseIds` | `courseRepository.selectById()` 逐条加载课程信息 | 30门课 = **30次额外查询** |
| **L610–L619** | 遍历 `msCourses` 检查 enrollment | `courseEnrollmentRepository.selectOne()` 逐条 | 同 L195 模式 |
| **L807–L869** | 遍历 `requiredCourses` 和 `electiveCourses` | L808 + L835 各一次 `selectOne()` ×2 每次迭代 | 30必修+20选修×2 = **100次额外查询** |

**性能影响**（100并发，微专业30门课场景）：
- N+1 总额外查询：~160次
- 每次 DB 往返：~2ms（本地 PG）
- 额外总延迟：~320ms/请求
- 100并发连接池峰值：可能瞬间消耗大量 HikariCP 连接

**修复建议**：
```java
// 循环前批量预加载
List<Long> allCourseIds = requiredCourses.stream()
    .map(MicroSpecialtyCourse::getCourseId).collect(Collectors.toList());
Map<Long, CourseEnrollment> enrollmentMap = courseEnrollmentRepository
    .selectList(new LambdaQueryWrapper<CourseEnrollment>()
        .in(CourseEnrollment::getCourseId, allCourseIds)
        .eq(CourseEnrollment::getUserId, userId))
    .stream()
    .collect(Collectors.toMap(CourseEnrollment::getCourseId, e -> e));
// 循环中用 Map.get() 代替 selectOne()
```

## 二、慢 SQL 候选（R10-慢查询）

### 2.1 🔴 P1-C：`selectCount(null)` 全表 COUNT（7处）

**文件**：`AdminStatsServiceImpl.java`

| 行号 | 调用 | 表 | 风险 |
|------|------|----|------|
| L107 | `courseRepository.selectCount(null)` | courses | 中等 |
| L115 | `enrollmentRepository.selectCount(null)` | enrollments | **高** — 选课表可能最大 |
| L118 | `videoRepository.selectCount(null)` | videos | 中等 |
| L121 | `exerciseRepository.selectCount(null)` | exercises | 中等 |
| L124 | `discussionPostRepository.selectCount(null)` | discussion_posts | 中等 |
| L131 | `certificateRepository.selectCount(null)` | certificates | 低 |
| L256 | `learningProgressRepository.selectCount(null)` | learning_progress | **高** — 学习进度表可能最大 |

**性能影响**：
- PostgreSQL 中 `COUNT(*)` 无 WHERE = **全表 Seq Scan**
- 10万行表：~50ms
- 100万行表：~500ms
- 7 处串行执行：最坏情况 **3.5秒**（100万行级别）
- 且 **不含 `deleted_at IS NULL`** 过滤，统计数据会包含已逻辑删除的记录

**修复建议**：
```java
// 方案 A：加删除过滤（最低成本）
courseRepository.selectCount(new LambdaQueryWrapper<Course>().isNull(Course::getDeletedAt));

// 方案 B：大表用 PostgreSQL 统计估算（高性能，可接受误差）
@Select("SELECT reltuples::bigint FROM pg_class WHERE relname = 'enrollments'")
Long estimateEnrollmentCount();

// 方案 C：Redis 缓存 + 定时刷新（5分钟 TTL）
```

## 四、连接池与服务器配置（R10-配置）

### 4.1 HikariCP 连接池

```yaml
spring.datasource.hikari:
  minimum-idle: 20
  maximum-pool-size: 250
  idle-timeout: 300000        # 5 min
  max-lifetime: 1800000       # 30 min
  connection-timeout: 5000    # 5 sec
  leak-detection-threshold: 60000   # 60 sec
  validation-timeout: 3000    # 3 sec
```

| 参数 | 当前值 | 评估 | 建议 |
|------|--------|------|------|
| `maximum-pool-size` | **250** | ⚠️ P1-I：单实例 250 连接过于激进 | 生产建议 ≤ 50（多实例部署按 `(core_count*2)+effective_spindle_count` 公式） |
| `minimum-idle` | **20** | ⚠️ P2：预分配 20 空闲连接浪费资源 | 建议 5~10 |
| `connection-timeout` | 5s | ✅ 合理 | — |
| `leak-detection-threshold` | 60s | ✅ 合理 | — |

**PostgreSQL 默认 `max_connections=100`**，本地开发环境 250 连接池实际无法全部建立。生产多实例部署时：3实例×50 = 150 已超。

### 4.2 Redis Lettuce 连接池

```yaml

  max-active: 16
  max-idle: 8
  min-idle: 2
  max-wait: 1000ms
```

✅ 配置合理，适合中小规模。

### 4.3 Tomcat / 服务器配置

```yaml
server:
  port: ${SERVER_PORT:8080}
```

**无自定义 Tomcat 线程配置**，使用 Spring Boot 默认值：

| 参数 | 默认值 | 评估 |
|------|--------|------|
| `server.tomcat.threads.max` | 200 | ⚠️ P2：200 线程 + 250 DB 连接 = 潜在 MySQL 连接风暴 |
| `server.tomcat.threads.min-spare` | 10 | ✅ |
| `server.tomcat.accept-count` | 100 | ⚠️ P2：突发流量排队深度可能不够 |
| `server.tomcat.max-connections` | 8192 | ✅ |

**P1-I 建议**：显式配置 Tomcat 线程池：
```yaml
server:
  tomcat:
    threads:
      max: 50
      min-spare: 10
    accept-count: 200
    max-connections: 1000
```

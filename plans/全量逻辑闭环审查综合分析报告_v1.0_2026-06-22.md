## 一、执行摘要

平台功能骨架已基本成型（P0 82% 实现、49 前端页面 100% 齐全、43 个 Controller 分层规范），**架构正确性不是主要问题，"逻辑闭环的守护机制"才是致命短板**。三个最重要发现：(1) **CI 从不执行 `mvn test`**（`ci.yml:29` backend-build 仅 `mvn compile`），导致 11 个已写好的测试（含 3 个 P0 并发测试）全部腐烂为"死测试"，制造**虚假安全感**；(2) **对象级授权碎片化**——无统一 `PermissionEvaluator`（`SecurityConfig.java:30` 仅 `@EnableMethodSecurity`），归属校验靠 Controller/Service 手写散落，大量端点以 `@PreAuthorize("isAuthenticated()")` 兜底，IDOR 防护"东补一处西补一处"；(3) **选课状态机无枚举、魔法字符串散落 6+ 处、核心状态实体无乐观锁**，并发下状态可错乱且无测试守护。**建议下一步：立即启动 Phase A（修 CI 屏障 + 安全止血），此为一切后续修复的前置闸门，不可与其他工作并行抢占。**

## 三、P0 缺陷清单（按风险叠加度排序）

> 叠加度 = 同一根因在多维度同时放大的程度。排第一者放大其余全部。

### P0-1 · CI 不执行 `mvn test`（叠加度 ★★★★★｜测试&CI / 后端 / DB）
- **描述**：CI 仅编译不测试，所有单测/集成测试/并发测试在合并门禁中被完全跳过。
- **维度**：测试&CI（根因）+ 后端 + 数据库（放大全部正确性缺陷）。
- **文件:行号**：`.github/workflows/ci.yml:29`（`mvn compile -B`）；`ci.yml:18-31`（backend-build 无 test step）。
- **触发条件**：任何 PR/push 合并——测试腐烂不被发现，回归无网。
- **修复方案**：backend-build 增加测试阶段（先止血再加严）：
  ```yaml
  - name: Unit & Integration Tests
    run: mvn -B verify           # 含 surefire(单测) + failsafe(*IntegrationTest)
    working-directory: micro-course-api
  ```
  止血期可先 `mvn -B test -Dtest='!*P0ConcurrencyTest'` 排除已损坏用例，待 P0-4 修复后移除排除项。

### P0-2 · 选课状态机三重失守（叠加度 ★★★★★｜后端L5 + DB + 测试）
- **描述**：状态用裸字符串、无枚举、无流转校验、无乐观锁、并发测试不跑。
- **维度**：后端（状态）+ 数据库（并发）+ 测试&CI（无守护）。
- **文件:行号**：`EnrollmentServiceImpl:114/138/451`、`entity/Enrollment.java`（无 `@Version`）、`EnrollmentP0ConcurrencyTest`（存在但不跑）。
- **触发条件**：并发重复选课/取消、非法状态跃迁（如 CANCELLED→ENROLLED）。
- **修复方案**：引入 `EnrollmentStatus` 枚举集中状态值，在 Service 加流转白名单校验，并给实体加乐观锁：
  ```java
  public enum EnrollmentStatus { ENROLLED, COMPLETED, CANCELLED;
    private static final Map<EnrollmentStatus,Set<EnrollmentStatus>> NEXT = ...;
    public boolean canTransitionTo(EnrollmentStatus t){ return NEXT.get(this).contains(t);} }
  // entity/Enrollment.java
  @Version private Integer version;
  ```

### P0-3 · 缺 PermissionEvaluator，对象级授权碎片化（叠加度 ★★★★☆｜后端L1+L3 + 文档）
- **描述**：无统一对象级授权机制，owner 校验散落且部分端点仅 `isAuthenticated()` 兜底。
- **维度**：后端（架构+接口）+ 文档契约（与权限矩阵脱节）。
- **文件:行号**：`SecurityConfig.java:30`（仅 `@EnableMethodSecurity`，无 `PermissionEvaluator` Bean）；兜底端点见 `DiscussionCommentController:25-48`、`VideoBookmarkController:29-42`、`ExerciseRecordController:44-72`。
- **触发条件**：登录用户携带他人资源 ID 访问/修改个人数据（IDOR）。
- **修复方案**：两步走——先给兜底端点补 Service 层 owner 校验（止血，参照 `EnrollmentController:135` 已修复模式），再实现 `PermissionEvaluator` 统一 `@PreAuthorize("hasPermission(#id,'Enrollment','OWNER')")`。

### P0-4 · 11 个测试腐烂 / 6 个已损坏（叠加度 ★★★★☆｜测试&CI）
- **描述**：测试代码已写但长期不执行，编译/断言失效无人知。
- **文件:行号**：`src/test/java/com/microcourse/` 下 `Auth/Course/Video/EnrollmentIntegrationTest`、`*P0ConcurrencyTest`、`VideoUploadP0ErrorTest`、`AsyncConfigP0ResourceTest`、`BaseIntegrationTest`。
- **触发条件**：随 P0-1 修复后，CI 第一次跑测试即红。
- **修复方案**：一次性修复 6 个损坏用例使其能编译通过，纳入 `mvn verify`，作为 P0-1 的配套。

### P0-5 · 30 个端点未实现 + 前端绕路风险（叠加度 ★★★★☆｜接口 + 前端 + 安全）
- **描述**：权限矩阵列出但后端缺失的 30 端点，前端可能绕路调用相邻端点。
- **维度**：后端接口 + 前端 + 安全。
- **文件:行号**：权限盘点报告"未实现端点清单"段。
- **修复方案**：按权限矩阵补齐缺失端点（带正确 `@PreAuthorize`），前端同步对账调用路径。

### P0-6 · LearningProgress 无乐观锁致进度并发错乱（叠加度 ★★★☆☆｜后端L5 + DB + 测试）
- **文件:行号**：`entity/LearningProgress`（无 `@Version`）、`LearningProgressController:77-84`、`LearningProgressP0ConcurrencyTest`（不跑）。
- **触发条件**：同一用户多端并发上报进度。
- **修复方案**：实体加 `@Version`，更新走 CAS；将并发测试纳入 CI。

### P0-7 · 通知矩阵未接线（叠加度 ★★★☆☆｜模块 + 文档）
- **描述**：通知基础设施齐全但业务事件零触发，用户收不到任何系统通知。
- **文件:行号**：`NotificationServiceImpl.java`（仅被 `EnrollmentServiceImpl:59` 间接引用查询，无主动 send 调用链）。
- **修复方案**：在选课成功/作业批改/审批等事件点注入 `notificationService.send(...)`，建立事件→通知映射表。

### P0-8 · `/api/files/**` 全量 permitAll 潜在越权读取（叠加度 ★★★☆☆｜后端L3 + 安全）
- **文件:行号**：`SecurityConfig.java:74`（`requestMatchers("GET","/api/files/**").permitAll()`）。
- **触发条件**：通配放行覆盖非封面的私有文件时，未授权读取。
- **修复方案**：收窄为封面专用前缀（如 `/api/files/covers/**`），其余文件改 `authenticated()` + 归属校验。

### P0-9 · ~49% 权限注解与权限矩阵 v2.0 不一致（叠加度 ★★★☆☆｜接口 + 文档）
- **文件:行号**：权限盘点"注解 vs 矩阵差异表"；样本如 `GradeController:37`（`hasRole('STUDENT')`）需对账。
- **修复方案**：逐端点对账，以权限矩阵 v2.0 为准修正注解，差异登记 `冲突评审决议.md`。

### P0-10 · 5 项 Phase 5-10 P0 功能缺口（叠加度 ★★☆☆☆｜后端 + 前端）
- **文件:行号**：Phase 5-10 实现度盘点"5 项缺口"段。
- **修复方案**：按 `docs/开发规划/phase5-10-spec.md` 补齐实现文件并过验收标准。

### P0-11 · 选课状态魔法字符串无单一真相（叠加度 ★★☆☆☆｜DB模块 + 后端）
- **文件:行号**：`EnrollmentServiceImpl:114/138/273/451`、`SlideController:143`、`OrderServiceImpl:87`、`NotificationServiceImpl:59`。
- **修复方案**：随 P0-2 枚举落地，全局替换字符串字面量。

### P0-12 · 3 项"部分实现"功能边界未闭环（叠加度 ★★☆☆☆｜接口 + 状态）

## 五、P2 缺陷清单（按工程改进排序）

1. **状态字段统一为 `Integer` vs `String` 不一致**（Course/Video 用 Integer，Enrollment/Order 用 String）——规范化。
2. **缺测试覆盖率门禁**（jacoco + 阈值）——`mvn verify` 后应加覆盖率 gate。
3. **smoke-test.sh 与 JUnit 集成测试职责重叠**——合并收敛。
4. **前端常量/枚举未与后端共享**——建议生成式契约同步。
5. **A11Y 自动化检查未入 CI**（playwright 可挂 axe-core）。
6. **PERF 预算未基线化**（无 Lighthouse/bundle-size 门禁）。
7. **`CreateTestPpt.java` 工具类混在 test 包**——应移至 fixtures。
8. **权限注解可读性差**（散落 SpEL），引入 `@PreAuthorize` 常量化或自定义注解。
9. **状态机文档与代码无双向校验脚本**——可加 convergence-check 扩展。
10. **大量 `isAuthenticated()` 端点缺统一审计日志**——对照 `OperationLogController`。
11. **CORS `allowedHeaders('*')`**（`SecurityConfig.java:93`）——可收窄白名单。
12. **`@Disabled`/损坏测试无标记策略**——建立 quarantine 机制。

## 六、修复路线图（3 阶段）

### Phase A · 立即修复（24–48 小时内）— 安全闸门 + CI 屏障
**目标**：堵住"无回归网 + 数据越权"两个致命口子。所有后续修复都依赖 A 先落地。
- 必修 P0：**P0-1（CI 跑 test）→ P0-4（修 6 损坏测试）→ P0-3 止血段（兜底端点补 owner 校验）→ P0-8（收窄 /api/files）**。
- **顺序硬约束**：P0-1 与 P0-4 必须同批提交（否则 CI 一开即红、阻塞所有 PR）。
- **验收标准**：
  - `mvn verify` 在 CI 绿；测试用例 0 编译错误、0 红。
  - 兜底端点新增越权访问测试（携带他人 ID → 403）全部通过。
  - `/api/files/**` 越权读取用例返回 401/403。

### Phase B · 一周内修复 — 业务正确性 + 核心链路测试
**目标**：保证选课/进度/通知三条主链路逻辑闭环且可验证。
- 必修 P0/P1：**P0-2（选课状态机+乐观锁）、P0-6（进度乐观锁）、P0-7（通知接线）、P0-5（30 端点）、P0-9（注解对账）、P1-1（5 核心链路集成测试）**。
- **顺序**：P0-2/P0-6（状态&并发底座）→ P0-7（通知依赖状态事件）→ P0-5/P0-9（接口对账）→ P1-1（在稳定底座上写测试）。
- **验收标准**：
  - 3 个 P0 并发测试在 CI 稳定通过；非法状态跃迁被拒。
  - 5 大核心链路集成测试纳入 `mvn verify`，覆盖率 ≥ 40%（核心包）。
  - 权限注解与矩阵 v2.0 差异清零并登记 `冲突评审决议.md`。

### Phase C · 两周内修复 — 状态机完整性 + 前端体验 + 文档同步
**目标**：补齐其余 6 个状态机、提升前端质量、契约回写。
- 必修 P1/P2：**P1-5（其余状态机）、P1-2/P1-3（A11Y/PERF）、P0-3 根治段（PermissionEvaluator）、P0-10/P0-12（功能缺口）、P2 全量**。
- **顺序**：先 PermissionEvaluator（替换止血补丁）→ 状态机补全 → 前端 A11Y/PERF → 文档 + 覆盖率/性能门禁。
- **验收标准**：
  - 7 状态机全部有枚举+流转校验+测试。
  - A11Y ≥ 90%、PERF ≥ 80%；jacoco/axe-core/lighthouse 门禁入 CI。
  - 状态机文档、权限矩阵、数据字典三方对账通过 convergence-check。

## 七、跨维度关联分析（缺陷放大效应）

### 案例 1 · 选课状态机缺失 × 无乐观锁 × 测试不跑 = 静默数据腐烂
- **单独风险**：状态枚举缺失=可读性差（可容忍）；无乐观锁=偶发并发问题；测试不跑=无回归网。
- **三重叠加**：并发选课/取消产生非法状态（CANCELLED→ENROLLED）写入 DB，**无乐观锁拦截、无状态校验拒绝、无测试报警**——脏数据静默累积，直到业务统计错乱才被发现，且无法定位何时引入。
- **一次性修复**：P0-2 一并落地 `EnrollmentStatus` 枚举 + 流转白名单 + `@Version` + 把 `EnrollmentP0ConcurrencyTest` 接入 CI（P0-1）。一个 PR 同时关闭后端L5/DB/测试三个红格。

### 案例 2 · 测试覆盖 ~3% × CI 不跑 mvn test = 测试债务无限累积 + 虚假安全感
- **单独风险**：覆盖率低=信心不足；CI 不跑=门禁松。
- **叠加**：11 个测试**已写但腐烂**，团队误以为"有测试保护"，实际每次合并都在裸奔；新增测试也立即腐烂——**债务复利**。这是比"没测试"更危险的状态。
- **一次性修复**：P0-1 + P0-4 同批——开启 `mvn verify` 的瞬间修复存量损坏用例，止住复利，并加覆盖率门禁（P2-2）防新增腐烂。

### 案例 3 · 30 端点未实现 × 缺 PermissionEvaluator × isAuthenticated 兜底 = 前端绕路 + IDOR 复合
- **单独风险**：端点缺失=功能不全；兜底鉴权=授权偏松。
- **叠加**：前端遇到未实现端点会**绕路调用相邻 `isAuthenticated()` 端点**，而这些端点缺统一对象级授权，携带他人 ID 即可越权——**功能缺口直接转化为安全漏洞**。
- **一次性修复**：P0-5 补端点时，统一通过 `PermissionEvaluator`（P0-3 根治段）声明 owner 校验，前端同步对账路径——一处机制同时关闭接口缺口与 IDOR。

## 八、回归验证方案

**Phase A 验收回归**
- 自动化：`mvn verify`（surefire+failsafe）全绿；新增"越权访问→403"用例集（Enrollment/Progress/Bookmark/Comment/Files）。
- 手动：用 B 用户 token 访问 A 用户资源 ID（取消选课、改进度、读私有文件）→ 全部拒绝。
- 数据完整性：无脏状态写入校验脚本。
- 性能：基线无回归（A 不引入性能改动）。

**Phase B 验收回归**
- 自动化：3 个 P0 并发测试稳定绿；5 核心链路集成测试通过；非法状态跃迁拒绝用例通过；核心包覆盖率 ≥ 40%。
- 手动：并发选课/取消压测（10 并发）状态一致；触发选课成功→收到通知。
- 数据完整性：`enrollment.status` 仅出现合法枚举值；`EnrollmentHistory` 审计完整。
- 性能：通知接线不阻塞主链路（异步发送）。

**Phase C 验收回归**
- 自动化：7 状态机流转测试全绿；axe-core A11Y ≥ 90%；Lighthouse PERF ≥ 80%；convergence-check 三方契约对账通过。
- 手动：键盘全流程可达；PermissionEvaluator 覆盖所有对象级端点抽检。
- 性能：bundle-size 门禁通过、首屏达标。

## 九、总工程师决策建议（有倾向性）

**总建议：立即启动 Phase A，且 Phase A 必须串行先行，不与 B/C 抢占。** 理由：CI 屏障（P0-1/P0-4）是所有后续修复的"测量仪器"，仪器未装就改业务=盲改。A 完成后，B 与 C 的前端部分（A11Y/PERF）可并行。

- **决策点 1 — A+B 并行 vs 串行**：**A 先串行（24–48h），B 紧随。** 不建议 A/B 并行：B 的状态机/通知改动若在 CI 跑测试之前合入，无法验证、风险叠加。A 完成后 B 内部各项可适度并行（状态底座→通知→接口）。
- **决策点 2 — 测试债一次性还清 vs 滚动**：**两段式——存量止血一次性，增量滚动。** P0-1+P0-4 一次性修复 6 个损坏测试并开启 `mvn verify`（止住复利）；5 大核心链路测试随 Phase B 滚动补齐。不建议一次性补全所有覆盖率（拖慢安全修复）。
- **决策点 3 — 引入 PermissionEvaluator vs 打补丁**：**引入，但分两步。** Phase A 先用 Service 层手写 owner 校验止血（参照 `EnrollmentController:135` 已验证模式，快、低风险）；Phase C 重构为统一 `PermissionEvaluator`（根治碎片化）。不建议 A 阶段就上 PermissionEvaluator——重构面大、阻塞止血速度。
- **决策点 4 — v1.1.0 前完成 vs 接受技术债**：**Phase A 必须 v1.1.0 前完成（安全+CI 不可妥协）；Phase B 强烈建议纳入 v1.1.0；Phase C 可接受滚动至 v1.2.0。** 即 v1.1.0 的发布闸门 = Phase A 全绿 + Phase B 核心链路测试通过。

**一句话执行指令**：先合一个"CI 跑测试 + 修 6 损坏测试 + 兜底端点 owner 校验 + 收窄 /api/files"的止血 PR，绿了再动业务。

## 十、附录

### A. 风险等级定义
- 🔴 **高**：直接导致数据错误、安全漏洞或回归不可控；阻塞发布。
- 🟡 **中**：影响正确性/体验/可维护性，可控但需排期。
- 🟢 **低**：健康或仅工程优化空间。

### B. P0/P1/P2 划分标准
- **P0**：安全漏洞 / 数据完整性 / 回归网缺失 / 阻塞合并门禁——24h–1 周内必修。
- **P1**：业务正确性与核心体验，影响用户主流程——1–2 周内修。
- **P2**：工程规范、可维护性、长期质量——滚动改进。
- 排序口径：P0 按"风险叠加度"（跨维度放大）；P1 按业务影响；P2 按工程收益。

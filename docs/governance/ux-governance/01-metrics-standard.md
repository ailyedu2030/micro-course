# 01 · 用户体验量化标准 v1 (UX Metrics Standard)

> **生效日期**：2026-07-21
> **覆盖范围**：微课管理平台全部前台系统（学员端 / 教师端 / 管理端）
> **目的**：把"用户感受"变成可观测、可比较、可追责的数字
> **状态**：v1（草案），首次复盘会后冻结为 v1.0

---

## 1. 标准的五个维度

> 用户的主观体验可以拆为五类客观可测信号。**任一维度失守，体验都会崩坏**。

```
维度 A  加载性能      —— 用户能否"立刻"看到东西？
维度 B  交互流畅度    —— 用户的每一下点击/输入是否被及时响应？
维度 C  易用性        —— 用户能否在 ≤3 次点击内完成任务？
维度 D  视觉舒适度    —— 用户的眼睛和脑子是否舒服？
维度 E  故障恢复速度  —— 出问题后多久能"自动回来"？
```

| 维度 | 子指标 | 阈值（绿/黄/红） | 采集来源 |
|------|--------|----------------|---------|
| **A 加载性能** | FCP（First Contentful Paint） | ≤1.5s / 1.5-3s / >3s | RUM（前端上报） |
| | LCP（Largest Contentful Paint） | ≤2.5s / 2.5-4s / >4s | RUM |
| | TTI（Time to Interactive） | ≤3.5s / 3.5-6s / >6s | RUM + Lighthouse |
| | 首屏 JS 体积 | ≤250KB / 250-500KB / >500KB | build 报告 |
| | 首屏 API 总耗时 | ≤1.0s / 1.0-2.5s / >2.5s | Grafana p99 |
| **B 交互流畅度** | 点击到反馈延迟（INP） | ≤200ms / 200-500ms / >500ms | RUM |
| | API 客户端感知延迟 | ≤800ms / 800-2000ms / >2000ms | Grafana p95 |
| | 滚动掉帧率 | 0-5% / 5-15% / >15% | 性能面板 |
| | 富文本编辑（Quill）操作往返 | ≤150ms / 150-400ms / >400ms | 前端事件戳 |
| **C 易用性** | 任务完成率（核心 8 任务） | ≥95% / 85-95% / <85% | 可用性测试 |
| | 关键路径点击数 | ≤3 / 3-5 / >5 | 流程 SOP |
| | 主操作可被搜索直达 | ≥80% / 60-80% / <60% | 站内检索埋点 |
| | 老年人完成率（≥60 岁） | ≥80% / 60-80% / <60% | 老年调研 |
| | 错误恢复率（点错能撤回） | ≥95% / 85-95% / <85% | 错误日志 + UX 走查 |
| **D 视觉舒适度** | 主题对比度（正文文字/背景） | ≥7:1 WCAG AAA / 4.5-7:1 AA / <4.5:1 FAIL | axe-core |
| | 字号基准（正文） | ≥16px / 14-16px / <14px | 设计 token 检查 |
| | 闪烁 | 无 / 偶尔 / 频繁 | axe-core |
| | 屏幕阅读器朗读合理率 | ≥95% / 85-95% / <85% | NVDA + VoiceOver |
| **E 故障恢复速度** | 致命错误恢复时间 | ≤5min / 5-30min / >30min | 应急剧本日志 |
| | 降级方案可用率 | 100% / 80-100% / <80% | 灰度开关面板 |
| | 用户回访率（故障后 24h） | ≥90% / 70-90% / <70% | Funnel |

## 2. 颜色 = 行动

> 阈值不是为了打分，而是为了**触发行动**。

| 颜色 | 含义 | 必触发动作 |
|------|------|---------|
| 🟢 绿 | 健康 | 持续监控 |
| 🟡 黄 | 警告 | 周复盘会上作为议程项目，由对应 owner 给出 7 天内整改计划 |
| 🔴 红 | 失败 | 24h 内责任人通报 + 4h 修复或灰度 | （见 05-incident-response.md）|

## 3. 采集方案（最小可起步）

### 3.1 必须有的数据源
1. **Grafana 看板（已有）**：把错误率 / 接口 p95 / p99 全部接入体验看板
2. **RUM（Real User Monitoring）**：前端在 `micro-course-admin/src/utils/rum.js` 中初始化，采集 FCP / LCP / INP / 点击错误
3. **Lighthouse CI**：每次 PR 必跑，并把分数回写到 PR 评论区
4. **axe-core**：每个 PR 必跑 a11y 扫描，FAIL 不予合并

### 3.2 采集点（建议清单）
- 全局错误捕获（已用 Sentry 风格的方案）
- API 调用链（已有 micrometer）
- 前端性能：FCP, LCP, CLS, INP, FID
- 富文本编辑器：操作往返、操作行数、Undo/Redo 比率
- 课件上传：渲染成功率、平均渲染时长（与 W37 压测报告挂钩）

### 3.3 不引入重量级 APM
> 微课项目目前不引入 DataDog / NewRelic 等付费 APM。
> 通过 **OpenTelemetry + Prometheus + Grafana + Loki** 免费栈完成即可。

## 4. 与 7 套治理动作的接口

| 治理动作 | 如何读这张标准 |
|---------|----------------|
| 每周复盘会 (02) | 复盘前必读红黄指标，作为会议议程起点 |
| 反馈 SLA (03) | 用户报怨须落到本表某行指标上，禁止泛"卡" |
| 无障碍基线 (04) | 维度 D 全部来自本基线 |
| 应急响应 (05) | 红色指标自动触发 |
| 健康度报告 (06) | 整张表 = 报告骨架 |
| 研究实验室 (07) | 实验基准就是这些指标的"前-后对比" |

## 5. 标准迭代流程

1. **季度回顾**：每个季度的最后一周复盘会上，逐项回顾阈值偏差，决定是否调整
2. **数据说话**：阈值调整必须有 ≥2 周的"为什么这一数字不合理"数据佐证
3. **变更记录**：每次改动都在 git log 与 CHANGELOG 中注明
4. **只改不删**：淘汰的旧指标保留在"历史阈值"小节，避免口径漂移

## 7. UX 攻坚 1 号动员令 · 10 项核心度量 (L0-UX-RALLY § 5)

> 启动 UX 攻坚后, 度量体系从"5 维"升级为"10 项核心度量 + 多源采集 + 常规复盘". 详见 [L0-MANIFESTO.md](./L0-MANIFESTO.md) § 二 #5.
> 关键变化: **NPS / SUS / 留存率 / 错误率**四项纳入 L0 治理, 与既有的 FCP / API p99 同等地位, 任一突破红线触发 P0-above.

### 10 项核心度量 (按公式 + 数据源 + 阈值 + 责任)

| # | 度量 | 公式 / 定义 | 数据源 | 阈值 (目标) | 责任 |
|---|------|------------|--------|------------|------|
| 1 | **FCP (First Contentful Paint)** | 首屏内容呈现时间 | RUM / web-vitals | ≤ 1.5 s | 前端 |
| 2 | **LCP (Largest Contentful Paint)** | 最大元素呈现时间 | RUM / web-vitals | ≤ 2.5 s | 前端 |
| 3 | **API p99 响应时长** | API 端点 99 分位 | monitor-lint | ≤ 800 ms | 后端 |
| 4 | **任务完成率** | (完成数 / 任务数) × 100% | 可用性测试报告 | 主任务 ≥95% / 边缘任务 ≥80% | UX + 客服 |
| 5 | **用户感知错误率** | (错误 session / 总 session) × 100% | Sentry / 前端日志 | ≤ 0.5% | 测试 + 运维 |
| 6 | **操作路径长度** | 完成主任务平均点击 / 步骤数 | 任务完成埋点 | 主任务 ≤ 3 步 / 边缘 ≤ 6 步 | UX + 前端 |
| 7 | **页面加载时长** | 全 DOMContentLoaded (含异步) | RUM | ≤ 3.5 s (3G 网络 ≤ 5 s) | 前端 |
| 8 | **NPS (Net Promoter Score)** | 推荐者% - 贬损者% | 月度回访问卷 | ≥ 30 | 产品 + UX |
| 9 | **SUS (System Usability Scale)** | 10 题问卷得分 × 2.5 | 月度回访问卷 | ≥ 70 | UX |
| 10 | **留存率** | (N 日后再次活跃 / D1 活跃) × 100% | 用户行为分析 | D7 ≥ 40% / D30 ≥ 20% | 客服 + 产品 |

> **注**: 1-7 是既有维度, 8-10 是 UX 攻坚新增. 每项有独立公式 + 单一数据源 + 阈值复审机制.

### 与既有"5 维"的关系

10 项核心度量是 5 维度的"操作化展开". 既有的 [§ 1 五维度](./01-metrics-standard.md) 不变, 本节新增在更高一层 (执行层). 任何变更必须先改 § 1 维度, 再映射 10 项.

### 失效红线 (新增口径)

- 任一红线条目连续 7 天不解决 = 触发 [EXECUTIVE-PRINCIPLE § 4](./EXECUTIVE-PRINCIPLE.md) 零容忍
- 月度报告缺任一红线条目 = 视为合规失约 (依 [EXECUTIVE-PRINCIPLE § 4 4 红线](../../EXECUTIVE-PRINCIPLE.md))
- 季度报告未复审 10 项阈值 = 月度健康度评分降级 [06 健康度报告模板 § 6](./06-health-report-template.md)

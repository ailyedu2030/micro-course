# 05 · 应急响应机制 · 1h 排查 / 4h 修复或灰度迭代

> **生效日期**：2026-07-21
> **适用范围**：影响 ≥5% 用户的体验危机
> **联动机制**：01 量化标准（红区触发）/ 03 反馈 SLA（P0 工单）/ 04 无障碍（视障/老年用户受阻即触发）

---

## 1. 应急响应等级

| 等级 | 触发条件 | 影响面估算 |
|------|---------|------------|
| **UX-P0** | 主任务不可达：登录/上传/播放，5%+ 用户失败；或 a11y 完全阻断某类残障用户 | >5% |
| **UX-P1** | 主任务劣化 60s+；或视觉/交互严重错位 | 1-5% |
| **UX-P2** | 二级功能异常，能通过旁路完成主任务 | <1% |

---

## 2. 时间窗（与 03 反馈 SLA 同步）

```
T+0:        警报触发 (AlertManager / Grafana 红区 + 工单 P0-C)
   ↓
T+1h:       首响应（必须），向 [值班群] 通告：
            - 影响面（数字）
            - 当前临时解决方案
            - 推定根因（首次）
            - 处理 owner
   ↓
T+4h:       决策二选一：
            (a) 修复并灰度（≤10% → 50% → 100%）
            (b) 灰度回滚到上个稳定版本（保留老功能）
   ↓
T+24h:      完整事故复盘 (Postmortem) 提交
            - 影响面精确数字
            - 时间线精确分钟
            - 直接根因 + 根本系统根因
            - 5 Whys
            - 防再发行动（每项有 owner + 截止时间）
   ↓
T+7d:       防再发行动全部落实 / 关闭
```

---

## 3. 角色与战时指挥

| 角色 | 战时职责 | 默认人选（轮值） |
|------|---------|----------------|
| **Incident Commander (IC)** | 总指挥，决定 4h 节点修复 vs 回滚 | 项目总负责人（默认）/ 运维值班（备用） |
| **Tech Lead** | 找根因 + 推动修复 | 后端/前端值班之一 |
| **Comms** | 对内通报 + 对外通告 | 产品值班 |
| **Recorder** | 时间线 / 决策 / 数据 | 测试值班 |
| **Recovery Owner** | 4h 节点的修复/灰度/回滚执行 | 运维值班 |
| **User Liaison** | 受影响用户回访 | 客服/外呼轮值 |

> **永远只有 1 个 IC**。IC 不写代码、不回用户。IC 只做决策与协调。

## 4. 应急剧本（按场景）

### 4.1 大面积白屏 / 接口 5xx

```
T+0:    AlertManager 触达 webhook
T+5m:   Tech Lead 确认非自身改动引起
T+15m:  Recorder 拉 Grafana 错误率 + 错误日志 tail
T+30m:  Tech Lead 给 IC 3 个候选根因 + 对应排查路径
T+45m:  IC 选 1 个，继续或开第二条
T+1h:   首响应 → 用户通告 "我们正在排查"
T+2h:   若需修复：分支建 hotfix
        若需回滚：执行 rollback（参考 ROLLBACK_PLAN.md）
T+4h:   修复 or 回滚上线；灰度 10% → 50% → 100%
T+6h:   全面恢复，发"已恢复"通告
T+24h:  Postmortem 提交
```

### 4.2 富文本编辑器异常（Quill 失败）

```
T+0:    内部测试发现 or 用户反馈到 P0-C
T+15m:  Recorder 上传控制台错误、抓取 DOM
T+30m:  Tech Lead 锁定到具体功能（菜单/上传/粘贴...）
T+1h:   通报 + 临时降级建议（"如遇工具栏丢失，请用键盘快捷键 Ctrl+B/I/U"）
T+2h:   hotfix PR（若简单）/ 灰度关闭新版（若复杂）
T+4h:   修复版或回滚到旧 textarea
T+6h:   上线
```

### 4.3 a11y 紧急失效（如棋盘色彩失效，对比度断裂）

```
T+0:    axe-core CI 红 / a11y 用户反馈
T+15m:  Tech Lead 确认范围（全局 / 单一组件）
T+1h:   通报"已发现 ... 正在回滚"
T+2h:   优先回滚视觉效果到上个稳定版
T+4h:   紧急修复或长期灰度
T+24h:  Postmortem 写"为什么没在 PR 阶段拦截"
```

### 4.4 老年用户外呼集中投诉

```
T+0:    客服收集到 ≥3 位 ≥60 岁用户同一问题投诉
T+2h:   产品值班 + 测试值班 现场陪外呼 1 小时
T+4h:   排查是否只在该人群触发
        (e.g. 字号 / 对比度 / 关键按钮位置 / 教学引导)
T+24h:  Postmortem + W37 富文本/HtmlBlock 类快速修复或回滚
```

## 5. 灰度开关（已部分实现）

> 项目现已具备灰度开关能力（按 PR #41 courseware_v2 实现）：
> - Feature Flag: `mc:feature:courseware_v2`（localStorage）
> - 服务端灰度名单（待补）：`mc:graylist:courseware_v2`
> - 回滚动作：通过 alert → 关 flag → 自上次工作台可用

> **新增**：未来需要为每个新功能预留灰度开关。

```js
// 标准灰度开关（草案）
{
  "flag_key": "mc:feature:audio_v2",
  "owner": "后端值班",
  "enable_default": false,
  "graylist_strategy": "user-id % 100 < 10",
  "rollback_action": "disable_flag + clear_localStorage"
}
```

## 6. Rollback 决策表

> 4h 节点时 IC 必须二选一：

| 选项 | 适用场景 | 风险 |
|------|---------|------|
| **修复 + 灰度** | 影响可定位，热修小，回归风险低 | 二次伤害可能 |
| **灰度回滚** | 影响难定位 / 用户已大规模撤离 / 复杂修复 | 老功能可能也有问题（需 pre-mortem）|

> **判断不通**：IC 越过 4h 节点不需更高审批，允许"再争取 2h"但必须发通告。

## 7. 通讯模板

### 7.1 对内（值班群）

```
[UX-INC P? - 工单 FB-xxxx]

T+0:   <触发描述，影响面数字>
T+1h:  <首响应已完成>
T+?:   <更新>
预计 4h 决策: <修复 or 回滚>
```

### 7.2 对外（用户通告，必要时）

```
关于 <事件> 的提示
我们刚刚发现 <场景>，会在 <T+?h> 内解决。
替代方案：<步骤>
客服热线：<24h 号码>
```

### 7.3 事后用户回访

```
<用户> 您好,
关于您反馈的 FB-xxxx，已在 v2.x 版本修复。
我们想确认您现在可以正常使用了吗？
—— 项目组
```

## 10. 失效红线

- UX-P0 未在 1h 内发出首响应
- IC 在 4h 仍未决策
- Postmortem 24h 内未出
- 防再发行动 7d 内未关闭
- 任何 critical 事故无 recorder 时间线记录

# 07 · 研究实验室 SOP · 灰度调研 / 实验室可用性测试 / 全场景压测

> **生效日期**：2026-07-21
> **目的**：在没有专职 UX 研究员的情况下，让工程团队也能跑出"用户语言"的结论
> **节奏**：每季度 1 次重点研究 + 每月 1 次小范围快测
> **联动**：01 量化标准 / 04 无障碍 / 05 应急响应 / 06 健康度报告

---

## 1. 三类研究形态

| 形态 | 频率 | 投入 | 产出 |
|------|------|------|------|
| **A 灰度调研（In-Field Study）** | 每季度 ≥1 轮 | 30-100 用户 | 真实使用数据 + 痛点清单 |
| **B 实验室可用性测试（Lab Usability）** | 每月 1 次 | 6-10 用户 | 任务完成率 + 任务时长 |
| **C 全场景压测（Stress Test）** | 每个 GA 版本前 | Locust + k6 | p95/p99 性能数字 |

> 三者配合 = "量化" + "质化" + "压力"。

---

## 2. A 灰度调研 SOP

### 2.1 准备
- **样本**：30-100 名真实用户（覆盖 K12 教师 / 学生 / 老年 / 职教学员）
- **设备**：覆盖 PC / Android / iOS / 老旧机型
- **期间**：1-2 周
- **入口**：通过应用内 banner 邀请（必须有 IRB-级别同意说明）

### 2.2 收集方式
- 应用内"反馈"按钮
- 每日活跃推送（"昨天用了什么？有不顺手吗？"）
- 客服外呼（老年专项）
- 自愿书面访谈（30 分钟）

### 2.3 必出指标
- 任务完成率（覆盖核心 8 任务）
- 任务平均完成时长
- 中途放弃率 + 放弃点截图
- NPS（推荐意愿）
- 老年/残障用户单独子集统计

### 2.4 报告
- 文档 commit 到：`docs/governance/ux-governance/research/YYYY-QM-in-field.md`
- 视频截图归档在 `docs/governance/ux-governance/research/YYYY-QM-in-field/`
- **不公开用户姓名/手机号/头像**

## 3. B 实验室可用性测试 SOP

### 3.1 流程
```
招募（6-10 人）  →  任务脚本  →  房间 / 远程  →  录像  → 复盘
```

### 3.2 必测 8 任务

| # | 任务 | 角色 |
|---|------|------|
| 1 | 登录账号，进入课程主页 | 全员 |
| 2 | 上传 PPT 课件 | 教师 |
| 3 | 编辑讲述稿 | 教师 |
| 4 | 用 AI 自动生成讲述稿 | 教师 |
| 5 | 生成 TTS 配音 | 教师 |
| 6 | 学生视角播放课件 | 学员 |
| 7 | 切换富文本/源码模式（HTML 编辑器） | 教师 |
| 8 | 提交反馈（应用内按钮） | 全员 |

### 3.3 评分维度（5 点）

| 维度 | 含义 |
|------|------|
| 任务完成 | 是 / 否 |
| 完成时长 | 秒 |
| 错误次数 | 含帮助请求 |
| 主观评分 SUS | System Usability Scale, 0-100 |
| 痛点词汇 | 自由描述后做语义聚合 |

### 3.4 报告
- 文档 commit 到：`docs/governance/ux-governance/research/YYYY-MM-lab-usability.md`
- 必含：每任务通过率柱状图 + 痛点高频词云 + 改进项对应 issue 链接

### 3.5 测试不是演戏
> **禁止**告诉用户"应该怎么点"。让用户自己想办法。
> 我们要收集的是"用户的路径"，不是"我们的设计是否聪明"。

## 4. C 全场景压测 SOP

### 4.1 工具
- Locust（已有 `micro-course/skills/.../scripts/load-test-million.py`）
- k6（如需要更细协议级，备选）
- 生产镜像（与生产同配置）

### 4.2 场景
- 浏览课程 50%
- 课件查询 30%
- 提交练习 10%
- 管理操作 5%
- 富文本/HTML 编辑 5%（长任务）

### 4.3 阈值（与 01 量化标准同步）
| 指标 | 阈值 |
|------|------|
| p50 | ≤ 100ms |
| p95 | ≤ 500ms |
| p99 | ≤ 2000ms |
| 错误率 | ≤ 0.1% |
| CPU | ≤ 70%（压测稳定期） |
| 内存 | ≤ 80% |
| DB connection | ≤ 70% 池大小 |

### 4.4 报告
- 文档 commit 到：`docs/performance/YYYY-MM-stress-test.md`
- 关联 PR 的 perf 数字对比
- 历史趋势曲线

## 5. 工具栈 + 流程约束

### 5.1 必备工具
- 录像 / 录音（用户授权）：OBS / 麦克风
- 屏幕镜像：Vysor / scrcpy / QuickTime
- 任务脚本：Markdown 模板 + 计时器
- 反馈归纳：Notion / MarkTag 或本仓库 issue 标签
- 数据分析：Python + pandas；可视化：Grafana 看板复用

### 5.2 流程纪律
- **IRB-级别同意**：开始前书面/口头同意；可随时退出
- **录像通知**：所有受试者明确知道录像
- **样本去重**：同一 UID 不连续参与
- **数据脱敏**：UID → 哈希；删除姓名/手机号/头像
- **结果匿名**：发布前去除个人识别信息
- **隐私合规**：符合《个人信息保护法》

## 6. 老年用户专项研究

> **强约束**：老年大学用户在产品定位中占比 35%，必须有专项研究 SOP。

### 6.1 招募
- 通过当地老年大学 / 社区中心 / 退休群（线下优先）
- ≥10 人 / 季度
- 年龄 60-75，性别均衡
- **不要求 IT 经验**

### 6.2 测试简化版
- 仅 4 个主任务：登录、找到课程、播放视频、上传作业
- 助手**只**在用户明确求助时回应
- 录像 + 关键错误记录

### 6.3 必查维度
- 完成率
- 完成时长
- 关键按钮是否被找到
- 客服电话是否被看到
- 字号/对比度反馈
- 语言是否清楚

### 6.4 必出改善
- 每月外呼节选反馈已存档至 `docs/governance/ux-governance/research/elderly/`

```

   ↓
```

## 10. UX 攻坚 1 号动员令 · 多维度用户研究 4 法 (L0-UX-RALLY § 3)

> 4 法定为 [L0-MANIFESTO.md](./L0-MANIFESTO.md) § 二 #3 落地操作, 由 UX 研究员 + 用户研究专员主导, R6 团队其他角色协同.
> **每季度必须跑满 4 法各 ≥1 轮**, 缺一即季度审计失约 (依 [EXECUTIVE-PRINCIPLE.md](./EXECUTIVE-PRINCIPLE.md) § 4).

### 4 法快表

| 法 | 目标 | 样本 | 工具 |
|----|------|------|------|
| **深度访谈 (Deep Interview)** | 挖出"用户没说出来的痛点" + 未被满足的期望 | 12-20 人/季度 (4 类客群各 3-5) | 半结构化脚本 + 录音 + 转译稿 |
| **焦点小组 (Focus Group)** | 验证需求方向, 多用户观点碰撞 | 4-6 组/季度, 每组 6-8 人 | 主持人脚本 + 录音 + 同意书 |
| **用户旅程地图 (Journey Map)** | 把"用户从首次接触到长期使用"分段, 标出触点 | 全客群全程 ≥3 个月覆盖 | Persona + 触点 + 情绪曲线 + 痛点-机会矩阵 |
| **竞品 UX 对标** | 找行业最佳实践 + 差异化机会 | 5-8 个同行 / 季度 | 评分表 (任务完成率 / 错误恢复 / a11y) |

### 触发节奏

- 每季度 1 次: 深度访谈 + 焦点小组 + 竞品对标 (3 周)
- 持续: 用户旅程地图 (滚动更新, 跨季度)
- 每个新功能前: 定向子集 (依 [03 §反馈 SLA](./03-feedback-sla.md))

### 产出物 (4 法各一份调研报告)

每份必须包含:
1. **背景**: 什么问题, 触发原因 (用户/竞品/产品)
2. **方法**: 4 法哪一法, 抽样, 工具
3. **发现**: 用用户语言 (引语 + 截图 + 视频)
4. **机会矩阵**: 痛点 × 频率 × 影响
5. **对应 issue 列表**: GitHub issue 链接 (依每条机会 ≥1 issue)
6. **负责人 + 时限**: 项目总负责人签字

### 报告存档

- 路径: `docs/governance/ux-governance/research/YYYY-MM-DD-{method}-{topic}.md`
- 加密: 录音 + 转译稿走 [../cases/](../cases/) 安全目录 (git-lfs 或加密 commit)
- 引用: 月度 / 季度报告必引当季 4 法报告链

### 4 法与既有治理的衔接

| 4 法产出 | 落到 |
|---------|------|
| 深度访谈痛点清单 | [feedback-pool.md](./feedback-pool.md) 工单化 |
| 焦点小组共识 | [01 量化标准](./01-metrics-standard.md) § 阈值复审 |
| 旅程地图 | [04 无障碍基线](./04-accessibility-baseline.md) § 触点清单复审 |
| 竞品对标 | [ROADMAP planning](../README.md) § 二 季度计划 |
| 全部 | [02 §7 一票否决清单](./02-weekly-review-ritual.md) 7 项否决项阈值复审 |

### 失效红线 (4 法专项)

- 单法人样本 <6 人 = 报告不计
- 单季度 4 法缺一 = 月度报告延期
- 单季度报告未 commit 或无 issue 链接 = 合规失败, 触发 [EXECUTIVE-PRINCIPLE § 4](../../EXECUTIVE-PRINCIPLE.md)

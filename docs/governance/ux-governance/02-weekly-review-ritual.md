# 02 · 每周用户体验专项复盘会 (Weekly UX Review Ritual)

> **频率**：每周 1 次，建议每周一 16:00（覆盖上周末数据 + 当周重点）
> **时长**：60 分钟（5 + 30 + 25 + 5 + 5）
> **召集人**：项目总负责人（UX 负责人）兼主持
> **参加人**：后端 / 前端 / 测试 / 运维 / 产品 / 客服 6 名值班（至少 4 名到会即可开会）
> **缺席影响**：值班连续缺席 2 次 = 视为放弃当周 SLA，下周自动换人

---

## 1. 会议结构（高度纪律化）

```
00:00 - 00:05  数据快照 (5 min)
   ├─ 5 维度指标上一周数字 (auto dashboard, 不需要人报告)
   ├─ 红色/黄色指标 anchor
   └─ 当周紧急事件时间线

00:05 - 00:35  案例深读 (30 min)
   ├─ 上周反馈池中选取 3 个真实案例
   ├─ 每个案例 8-10 分钟, 5 段拆解
   └─ 提问-回答-原因-修复-预防

00:35 - 01:00  行动决策 (25 min)
   ├─ 新 issue 评审 (≤ 5 个)
   ├─ 红色指标整改计划表决
   ├─ 下周排班确认
   └─ 6 名 owner 认领
```

### 1.1 数据快照

> **禁止 PPT**。直接从 Grafana 看板 + 反馈池拉数据。

输出：
- 5 维度指标红黄绿密度图
- 当周任何"红色"指标 = 本周唯一优先级
- 上周闭环情况：N 个开放 issue 是否已完成

### 1.2 案例深读（每个案例 5 段）

> **5 段结构**——这是在没有专职 UX 团队时也能运作的纪律。

```
1. 还原场景    —— "用户做了 X，想要 Y，结果 Z"
2. 实际触达点  —— 哪条 API / 哪个组件 / 哪个路由
3. 根因        —— 为什么走到这里
4. 行动        —— 谁、什么时候、改什么
5. 防止再发    —— 哪条治理动作需要更新
```

**只接受被记录在 [templates/feedback-ticket.md](./templates/feedback-ticket.md) 的真实反馈**。
> 不接受口头的、上周的、没有 UID 的"听说有人..."。

### 1.3 行动决策

每个开放 issue 必须满足：
- 责任人（单一 owner，禁止"团队"）
- 时限（≤ 7 天 / ≤ 14 天 / 灰度）
- 验证方式（用什么数据/实验/用户回访证实已修复）

## 3. 会后 24h 必产出

1. **会议纪要**：commit 到 `docs/governance/ux-governance/minutes/YYYY-MM-DD-weekly-ux-review.md`
2. **issue 同步**：所有行动项同步创建 GitHub issue，标签 `ux-review`
3. **看板刷新**：Grafana 看板写一条本周重点
4. **通知发送**：值班群发一份"本周 UX 三件事"摘要

## 4. 季度"组合复盘会"

> 每季度的最后一周复盘会升级为 **90 分钟"组合复盘"**：

- 回顾本季度 5 维度指标趋势
- 评估 W37 Phase 5 清理是否需要延期（与 dev governance 联动）
- 决定下一季度 UX 标准的阈值调整
- 评选"本季度最具用户体验改善价值的工程师"（内部认可）

## 6. 失效红线

> 这五条**任一**触发立刻开紧急会，不等一周：

- 任何维度进入 🔴 红区超过 24h 未责任人认领
- 24h 内累计 ≥ 5 条用户投诉指向**同一**功能
- 月度健康度评分连续两周下降
- 老年用户外呼中出现"无法完成主任务"的反馈
- 关键页 a11y axe-core 扫描失败

## 7. L0 UX 一票否决清单 (UX 攻坚 1 号动员令 § 6)

> 任何**项目节点** (Sprint 验收 / PR 合并 / 灰度发布 / 正式发布 / 月度复盘 / 季度审计) 需满足下列 7 项**一票否决**清单. 任一不达标 = 节点否决, 不可进入下一节点.

| # | 否决项 | 阈值 | 数据来源 | 责任人 |
|---|--------|------|----------|--------|
| 1 | 用户任务完成率 | ≥95% 主任务 + ≥80% 边缘任务 | 可用性测试报告 / 任务完成率 dashboard | UX 值班 |
| 2 | 页面/接口响应时间 | FCP ≤ 1.5s + API p99 ≤ 800ms | RUM / monitor-lint | 前端值班 + 后端值班 |
| 3 | 错误率 | 用户感知错误率 ≤ 0.5% | Sentry / 日志聚合 | 测试值班 + 运维值班 |
| 4 | 无障碍合规 | axe-core 0 critical / WCAG 2.1 AA 100% | a11y CI 报告 | UX 值班 |
| 5 | 老年/残障适配 | 老年字号 ≥18px / 屏幕阅读器走读通过 | a11y 专项测试 | UX 值班 |
| 6 | NPS / 满意度 | 上版本 NPS ≥30 + SUS ≥70 | 反馈池 + 月度回访 | 产品值班 + UX 值班 |
| 7 | 反馈池及时性 | 首响应 24h + 修复 72h (P1-C) | 反馈池统计 | 客服值班 |

### 否决流程

1. 任何值班可在节点前 24h 触发"一票否决"
2. 项目总负责人必须在 4h 内确认否决生效
3. 否决产生的延期 / 返工 = 节点负责方的可见成本, 不转嫁到 UX 维度
4. 否决原因 / 修复方案 / 重审时间 在 [feedback-pool.md](./feedback-pool.md) 持续跟踪

### 申诉通道 (避免误否决)

- 节点负责方可在 24h 内向总负责人申诉
- 总负责人召集 UX 值班 + 节点 owner + 关联值班 = 复审
- 复审结论: 维持否决 / 撤回否决 / 部分有条件放行
- 申诉过程与结论全部落入 [feedback-pool.md](./feedback-pool.md) 审计链

### 不允许的"绕过否决"行为

- ❌ "反正快完成了, 上线吧" → 节点须重审
- ❌ "明天补 a11y" → 节点须重审
- ❌ "用户不会用这个按钮" → 节点须重审

### 一票否决与铁律只令 4 (零容忍) 的联动

- 节点否决连续 3 次 = 触发 [EXECUTIVE-PRINCIPLE.md](./EXECUTIVE-PRINCIPLE.md) § 4 零容忍红线
- 违反者按 P0-above 处理
- 案例全部落入 `docs/incidents/YYYY-MM-DD-ux-veto-bypass.md`

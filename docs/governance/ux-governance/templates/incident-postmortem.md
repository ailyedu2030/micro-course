# 事故复盘 (Postmortem) 模板

> **用于**：UX-P0 / UX-P1 事故
> **发出时间**：T+24h 内
> **存放位置**：`docs/governance/ux-governance/incidents/YYYY-MM-DD-<short-title>.md`
> **阅读人**：全员；不公开个人追责

```markdown
# Postmortem · YYYY-MM-DD · <一句话标题>

| 字段 | 取值 |
|------|------|
| **事故等级** | UX-P0 / UX-P1 / UX-P2 |
| **触发时间** | YYYY-MM-DD HH:MM (UTC+8) |
| **恢复时间** | YYYY-MM-DD HH:MM |
| **首响应时间** | YYYY-MM-DD HH:MM (T+?) |
| **决策时间** | 修复 or 回滚 @ T+4h 或更晚 |
| **影响面** | X 用户 / Y 任务次数 |
| **Incident Commander** | @<name> |
| **Tech Lead** | @<name> |
| **Recorder** | @<name> |

## 摘要（≤150 字）
<写到第 3 行时，读者已知道：什么坏了, 影响多大, 怎么恢复的。>

## 1. 时间线（按分钟）
- T+0:    <事件>
- T+5m:   <响应>
- T+15m:  <诊断>
- T+1h:   <首次通报>
- T+2h:   <分支>
- T+4h:   <决策 + 修复/回滚>
- T+?:    <灰度进度>
- T+?:    <恢复>
- T+24h:  本文档提交

## 2. 影响面
- 受影响用户数
- 受影响主任务 / 操作
- 受影响设备 / 浏览器
- 受影响地区（老年 / 残障 / K12 ...）
- 性能指标偏差（p95/p99 上升多少，错误率涨多少）

## 3. 直接根因 (技术)
- <代码哪一行错了 / 配置哪个键错了>
- 触发条件（如：某次 PR 的某次提交）

## 4. 根本系统根因（5 Whys）

> **Why 1**: ...
> 因为 ...

> **Why 2**: ...
> 因为 ...

> **Why 3**: ...
> 因为 ...

> **Why 4**: ...
> 因为 ...

> **Why 5（系统性）**: ...

## 5. 行动总表

| 类型 | 内容 | owner | 截止 | 状态 |
|------|------|-------|------|------|
| 缓解 | 回滚 X 到上一个稳定版本 | @tom | T+4h | done |
| 调查 | 拉通监控... | @jerry | T+1d | done |
| 防再发 | 加 a11y 测试用例 | @alice | T+7d | open |
| 防再发 | 把变更流程加入 PR template | @bob | T+3d | open |
| 防再发 | 更新 01 量化标准的阈值 | @总负责人 | 下次复盘 | pending |

> 所有"防再发"行动 7 天内必须 close。

## 6. 教训（团队级）
- 我做错了什么（系统/流程）？
- 哪些治理动作本可以拦住？
- 哪些治理动作需要新增/修订？

## 7. 利好
- 哪些灰色动作被我们抓到？
- 哪些治理动作的有效性被验证？

## 8. 时间线证据
- Grafana 截图（事故窗口前后）
- 错误日志链接
- 录像 / 截图链接
- 用户反馈原文（脱敏）

---
> 本文档由 IC 提交到 incidents/，所有人有 review 权。
> 严禁追责个人。
> 7d 后无新评论视为定稿，移入 `incidents/closed/`。
```

---

## 质量红线

- 时间线精确到分钟
- 至少 5 Whys
- 防再发行动项每条必带 owner + 截止
- 7 天内全部 close，否则下一周复盘必须重提
- 不允许"缺少数据"，缺数据 = 紧急补 = 这本身是事故

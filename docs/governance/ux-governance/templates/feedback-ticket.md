# 用户反馈工单 · 模板

> **用于**：所有反馈渠道（应用内/电话/微信/GitHub/邮件/教室）都必须汇总为这个结构
> **保存位置**：`docs/governance/ux-governance/feedback-pool.md` 内一条
> **状态流转**：`new → triaged → in-fix → verified-by-user → closed / wontfix`

```markdown
## FB-<编号>

| 字段 | 取值 |
|------|------|
| **通道** | [in-app / phone / wechat / github / email / classroom] |
| **创建时间** | YYYY-MM-DD HH:MM (UTC+8) |
| **首响应时间** | YYYY-MM-DD HH:MM（≤SLA） |
| **处理 owner** | @<角色> <名字> |
| **状态** | new / triaged / in-fix / verified-by-user / closed / wontfix |
| **最终等级** | P0-C / P1-C / P1-R / P2-R / P3-R |

### 用户画像（脱敏）
- UID: `uid_hash(原UID)` 或 匿名
- 年龄段: <60+ / 30-60 / 18-30 / <18>
- 角色: 学员 / 教师 / 管理员 / 老年大学 / K12
- 设备: PC / Android / iOS, 浏览器型号
- 网络: 4G / 5G / Wi-Fi / 弱网

### 反馈内容
- **场景（一句话）**: 用户做了 X
- **期望**: 用户原本想要 Y
- **实际**: 实际发生 Z
- **触达点**: /path/... 或 组件名 + API endpoint
- **截图/录音**: [链接]

### 落到的体验指标（详见 01）
- 主: 维度 X / 子指标名
- 辅: ...

### 时间线
- T+0: 创建
- T+?: 首响应
- T+?: 定级
- T+?: 方案提交 (issue 链接)
- T+?: 修复合并
- T+?: 用户回访
- T+?: 关闭

### 复盘摘要（关闭后）
- **根因**: ...
- **整改方案**: ...
- **防再发**: ...

### 用户回访记录
- YYYY-MM-DD: 联系用户 <姓名脱敏>，用户反馈"已可正常使用"
```

---

## 使用规则

1. 每条反馈必须使用本模板**完整字段**录入
2. 缺失字段由产品值班在 24h 内补齐
3. Slack/IM 中不允许"只在群里提一句"，必须落到工单池
4. closed 状态必须包含"用户回访"或"wontfix 原因"
5. 季度复盘时统计所有 wontfix 反馈的合理性，决定是否调整需求优先级

---

## wontfix 决策模板

> 用于在反馈会上判定 wontfix 时填写：

- **为何不做**：<技术债务 / 长期演进 / 优先级冲突>
- **提供替代方案**：<已知可用方式>
- **是否破坏红线**：<WCAG / 主任务 / 老年场景，任意一项命中则不允 wontfix>
- **下次评审时间**：<N 季度后重新评估>
- **签字**：<产品值班 + 项目总负责人>

# UX 至上 · 项目铁律执行框架

> **签发人**：项目总负责人 / 总工程师
> **生效日期**：2026-07-21（与 UX 治理体系同日启动；本框架为该体系的"只令与执行"顶层文档）
> **效力**：高于 AGENTS.md P0 生产安全铁律，仅次于不可抗力（监管/合规法律要求）；与 AGENTS.md P0 互为正交层级
> **作用域**：项目全生命周期（调研 → 设计 → 开发 → 灰度 → 上线 → 运维 → 退市）+ 所有跨职能角色（产品/设计/开发/测试/运维）

---

## 〇、铁律只令（不可妥协 · 不可绕过）

### 只令 1：用户体验至上
- **在任何场景下都不将时间和成本作为限制条件**
- 任何工程决策的最终裁判准则：**该决策是否改善了至少一类用户的真实体验（可用性 / 可访问性 / 情感 / 信任 / 期待）**
- "成本/时间压力"在 UX 决策中**不可作为理由**；只能作为"次要约束"协商工时与人员规模

### 只令 2：用户体验唯一不可妥协
- 任何降级体验的权宜之计（缩减流程、阉割功能、加广告、限速、强制登录等）必须先经 UX 评审会评估；不得个人擅自决定
- 若 UX 评审认为不可妥协，则否决；由责任方承担返工成本
- 例外：当且仅当"法律合规要求"压倒 UX，才可破例

### 只令 3：所有节点考核指标以用户体验锚定
- 任何"项目节点"（sprint/milestone/release）的考核 KPI 默认口径 = **UX 评分**（NPS + 任务完成率 + 可用性系统可用分 SUS）
- 偏离 UX 优先的指标（进度数字、代码量、Coverage%）作为**辅助参考**，不得作为上线放行依据

### 只令 4：违反铁律零容忍
- 个人决策"为赶工而牺牲 UX" = 重大治理事故，按 P0-above 处理
- 项目负责人 / 总负责人 / 总工程师若示意默许此类行为，同等追责
- 任何角色可越级举报此类行为，受项目治理机制保护（匿名通道见 §4.4）

## 一、用户 5 条要求的落地执行框架（2026-07-21 启动即生效）

### 1. 全员对齐会议

**目标**：将铁律同步到每一位项目参与者的认知与行为，而非停留在文档。

| 动作 | 时点 | 责任 |
|------|------|------|
| 启动宣贯大会 (Launch-Day) | 启动当周 | 项目总负责人主持 + 全员必签《铁律接收确认函》 |
| 月度铁律回顾 (15 min in 每月月报头条) | 每月 1 日 | 月报作者（产品值班） |
| 入职铁律 onboarding | 新成员入职 D1 | 人事对接 + 总负责人 |
| 紧急铁律宪法更新会 | 宪法变更时召集 | 总负责人 |
| 工程文化墙 (Slack/Slack-Like) #ux-constitution 频道 | 持续 | 客服值班维护 |

**会议纪要存档**：`docs/governance/ux-governance/minutes/YYYY-MM-DD-ux-constitution-launch.md`，首次启动大会必产生 1 份 baseline。

### 2. 全流程 UX 保障机制（评审节点）

**目标**：把 UX 评审节点硬编码进流水线，不可跳过。

| 节点 | 强制评审人 | 不可进入下一阶段的判定 | 关联模板 |
|------|------------|----------------------|---------|
| 需求调研阶段 | UX 值班 + 客服值班 | 用户画像/场景缺失则返工 | — |
| 原型设计阶段 | UX 值班 + 产品值班 | 交互动线、合规可达性未通过则返工 | [a11y-acceptance-checklist.md](./templates/a11y-acceptance-checklist.md) |
| 功能开发阶段 | UX 值班 + 开发值班 | PR 未带 UX owner 字段则返回 | 见下文 PR template |
| 灰度测试阶段 | UX 值班 + 测试值班 + 客服值班 | 任务完成率 < 95% / 主任务未达则返工 | [03-feedback-sla.md](./03-feedback-sla.md) §4 KPI |
| 正式上线阶段 | 项目总负责人 + UX 值班 + 运维值班 | 健康度评分不达标则延后 | [06-health-report-template.md](./06-health-report-template.md) |
| 上线后运维阶段 | UX 值班 | 季度复盘发现黄/红指标则启动整改 | [02-weekly-review-ritual.md](./02-weekly-review-ritual.md) |

**PR Template 强制项**（已落地，下方落 commit）：
- [ ] UX 评审 owner 指定 (阻塞合并)
- [ ] a11y checklist 全部勾选 (阻塞合并)
- [ ] 端到端体验影响说明 (阻塞合并)

### 3. 充足专项资源保障

**目标**：用户的"体验不受时间和成本限制"必须落到具体资源承诺，而非口号。

| 类别 | 资源规格 | 配置位置 |
|------|---------|---------|
| 顶尖 UX/UI 设计 | ≥1 名全职 UX 设计师 + 1 备用顾问（合同） | rostering.md / HR |
| 用户调研池 | ≥10 名老年大学 + ≥10 名中职教师 + ≥10 名 6-12 年级学生 + ≥5 名视障/运动障碍 | [07-research-lab-sop.md](./07-research-lab-sop.md) §6 老年 / [07-research-lab-sop.md](./07-research-lab-sop.md) §7 残障 |
| 全链路用户行为分析 | RUM 数据: FCP / LCP / CLS / INP / FID (前端) + API p95/p99 (后端) | [01-metrics-standard.md](./01-metrics-standard.md) §1 维度 A/B + monitor-lint |
| 用户反馈多通道 | 应用内反馈按钮 + 客服电话 + 微信公众号 + 教室现场 | [03-feedback-sla.md](./03-feedback-sla.md) §1 反馈入口 |
| 月度回访预算 | 抽样 ≥30 用户/月 + NPS 季度结构化访谈 | [06-health-report-template.md](./06-health-report-template.md) §4 数据采集口径 |
| UX 应急预算 | 1 名 UX 工单应急支持 24h 可达 | rostering.md 应急联系人矩阵 |
| 设计与开发脱钩时间 | 设计与开发同等级并发节奏（开发不阻塞设计评审） | rostering.md |

> **总负责人承诺**：UX 资源缺口视为 P1 治理事件，优先资源倾斜填补。

### 4. 问责机制

**目标**：把铁律落到人头和组织考核，避免"纸面合规"。

| 维度 | 责任主体 | 处罚 |
|------|---------|------|
| 项目 KPI 偏离 UX | 项目总负责人 | 月度复盘必声明 + 次月整改计划 + 当季绩效扣分 |
| 单 PR 暗藏降体验 | PR 作者 + Reviewer | 强制 revert + 团队批评 + 个案回炉培训 |
| 跳过 UX 评审节点 | 当阶段 Owner | 关闭该 PR/工作流 + 复盘事故 + 个案回炉培训 |
| "为赶工违反铁律" | 决策者 | 零容忍（只令 4）— 治理 P0-above 处理 |
| UX 资源不足未上报 | 资源 Owner | 月度报告必声明 + 资源补充承诺 + 季度审计 |

**考核文本来源**：
- 月度报告 [06-health-report-template.md](./06-health-report-template.md)：用户体验评分作为头条
- 季度报告：NPS + 核心 UX KPI 与商业指标对齐展示（即使商业达标，UX 不达标也要标题报警）

**红线回顾**（与既有治理一致）：
- 任何 PR 不带 a11y 验收清单 → 不合并（[01-metrics-standard.md](./01-metrics-standard.md) § 启动即生效 4 红线）
- 任何用户反馈未录入 [feedback-pool.md](./feedback-pool.md) → 反馈 SLA 失效
- 任何 UX-P0 事故 24h 内无 Postmortem → 治理失守
- 任何月度健康度报告延期 ≥7 天 → 不允许签发

### 5. 定期复盘

**目标**：持续迭代，避免"一次性合规"。

| 频次 | 文档 | 输入 | 输出 |
|------|------|------|------|
| **每周** | [02-weekly-review-ritual.md](./02-weekly-review-ritual.md) | 上周反馈池 + 月度健康度 | 行动项 + owner + 7 日内整改 |
| **每月** | [06-health-report-template.md](./06-health-report-template.md) | 5 维度指标 + 反馈池 | 月度 UX 健康度评分 + 商业 KPI 校准 |
| **每季** | 季度合集（含月报 + UX 值班合规审计） | 季度累积数据 | 阈值调整提议 + Roadmap |
| **每年** | 宪法复审 | 全部治理文档 + 重大事故 | 修订或维持 |

**复盘纪律**：
- 每次复盘必须回答 4 个问题：(a) 哪些体验实测值好转？(b) 哪些变差？(c) 原因是系统还是个人？(d) 下期阈值/资源如何调整？
- 无答复 = 复盘无效，由 UX 值班主动标注"零答复" → 升级到总负责人

## 二、与既有治理的集成

### 2.1 与 [README.md](./README.md) 关系

UX 治理体系（含 01-07 + templates + 反馈池）已经在 2026-07-21 启动；本 EXECUTIVE-PRINCIPLE 是其"宪法级"补充，**不取代**既有规范，而是把"铁律 / 问责 / 资源"3 个高阶要素制度化。

### 2.2 与 AGENTS.md 关系

AGENTS.md P0 生产安全铁律（在 P0 层）继续生效；UX 铁律位于 **P0-above**（更高级别）：当遇到 P0 安全 vs UX 冲突时，**默认 UX 让 P0 安全**（如暗黑模式不能为了好看省略 CSRF / 鉴权）；当遇到 UX vs 时间成本冲突时，**默认 UX 胜**（铁律 1-3）。

### 2.3 与 PR / 模板关系

PR template 强制项：
1. UX 评审 owner 指定 (@required)
2. a11y checklist 全部勾选 (@required)
3. 端到端体验影响说明（如果有 PR 是体验降级，必须在 description 显式声明并由 UX owner 复议）

# 03 · 用户反馈快速响应通道 + SLA + 工单分级

> **生效日期**：2026-07-21
> **目标**：用户反馈从"听说"变成"在表格里" → 24h 定级 → 72h 出整改方案
> **负责人**：产品值班（轮值，由排班表决定）
> **协同**：客服/外呼轮值负责老年用户录音与转写，后端/前端值班负责技术判断

---

## 1. 反馈入口（多通道汇总）

> **所有反馈必须收口到工单池** `docs/governance/ux-governance/feedback-pool.md`，**禁止**任何反馈"只在群里提一句"。

| 通道 | 接入方式 | 字段自动填充 |
|------|---------|-------------|
| 应用内"反馈"按钮 | 前端 POST `/api/feedback/in-app` | 用户 ID、设备、当前页、网络 |
| 客服电话（老年用户） | 客服记录到工单池 | 人工转写（≤24h） |
| 微信客服号 | bot 抓取 | 微信 ID、人工分类 |
| GitHub Issue `ux-feedback` | 自动同步 | issue 内容 |
| 邮件 `ux@ailyedu.cn` | mail forwarder | 邮箱、时间戳 |
| 教室现场观察记录 | 老师自愿提交 | 教室 ID |

## 2. 工单结构（必须字段）

每个工单进入反馈池时强制结构化：

```
FB-编号:
  - 通道:           [in-app / phone / wechat / github / email / classroom]
  - 用户:           [UID 或 匿名, 加 年龄段 / 角色 / 设备]
  - 场景:           [一句话描述]
  - 触达点:         [页面/组件/API 路径]
  - 期望:           [用户原本想要发生什么]
  - 实际:           [实际发生了什么]
  - 截图/录音:      [链接]
  - 严重度初判:     [P0 / P1 / P2]
  - 创建时间:       [ISO]
  - SLA 截止:       [创建时间 + 24h / 72h]
  - 处理 owner:     [轮值角色 + 名字]
  - 状态:           [new / triaged / in-fix / verified-by-user / closed / wontfix]
```

模板见 [templates/feedback-ticket.md](./templates/feedback-ticket.md)

## 3. 分级标准（用户体验维度）

> 分级 ≠ bug 优先级。**用户体验问题有自己的分级标准**。

| 等级 | 定义 | 示例 |
|------|------|------|
| **P0-C** | 主任务不可达 + 5%+ 用户受影响 | 登录失败、白屏、课件完全无法播放 |
| **P1-C** | 主任务可达成但体验明显劣化 | 渲染等待 60 秒、工具栏丢失、按钮无响应 |
| **P1-R** | 主任务完成但用户**不再回来** | 噪音反馈、延迟感知到难以坚持 |
| **P2-R** | 美观/小瑕疵，不影响主流程 | 字号偏小、对齐差、动画略卡 |
| **P3-R** | 建议/优化，多数用户感知不到 | "希望能加个黑暗模式" |

> 区分 `P1-C` 与 `P1-R`：
> - `C`（Customer visible）= 用户**看到**错误
> - `R`（Retention）     = 用户**放弃**再使用
> 二者都属 P1，但 R 是更隐蔽的流失，需要重点跟踪。

---

## 4. SLA（处理时限）

| 等级 | 首响应（≤X h） | 定级（≤X h） | 整改方案（≤X h） | 修复（≤X d） | 用户回访（≤X d） |
|------|---------|---------|---------|---------|---------|
| **P0-C** | 1h | 4h | 24h | 4h（应急响应启动） | 1d |
| **P1-C** | 4h | 24h | 72h | 5d | 3d |
| **P1-R** | 4h | 24h | 72h | 10d | 5d |
| **P2-R** | 24h | 72h | 5d | 30d 或纳入 backlog | 7d |
| **P3-R** | 24h | 72h | 14d（评估窗口） | 下个 sprint | 14d |

> **首响应**：在反馈池里 @owner、回复用户"我们已收到"
> **定级**：产品值班根据分级标准 + 数据判断最终等级
> **整改方案**：落到 issue 标题为 `[P?C] 反馈 FB-xxxx 的整改方案`
> **修复**：代码合入 + 灰度 + 数据证明
> **用户回访**：被反馈者本人收到结果通知

## 5. 工单流转图

```
[入口]  → 反馈池一条结构化记录
  ↓
[产品值班 24h 内] → 定级 → 通知对应 owner
  ↓
[对应 owner 72h 内] → 创建整改 issue → 进入 GitHub Project
  ↓
[修复期内]  → status = in-fix → 每周复盘会上汇报进度
  ↓
[修复完成 + 灰度]  → status = verified-by-user
                       ↓
                     [关闭] 反馈者回访确认
```

> **回访率** = 已闭环反馈 / 总反馈 × 100%，目标 ≥ 90%。

## 7. KPI 计算（每月报给健康度报告）

| 指标 | 公式 | 目标 |
|------|------|------|
| **首响应达成率** | 首响应 ≤SLA / 总反馈 × 100% | ≥ 95% |
| **定级达成率** | 定级 ≤24h / 总反馈 × 100% | ≥ 90% |
| **方案达成率** | 方案 ≤72h / 已定级反馈 × 100% | ≥ 95% |
| **回访达成率** | 关闭含回访 / 总关闭 × 100% | ≥ 85% |
| **P0 平均首响应时长** | sum(首响应 - 创建)/n（仅 P0） | ≤ 30min |
| **回归率** | 关闭后再次被反馈 / 总关闭 × 100% | ≤ 5% |

---

## 8. 关键守则

1. **首响应 = 真的回复用户**，不只是看一遍。"我们正在处理，2 天内回复方案"——这就够了。
2. **永不丢失反馈**。即使归为 wontfix，也要在反馈池里写下"为什么放弃"和"我们提供了什么替代方案"。
3. **老年用户的反馈必须有电话回访**，不仅是文字回复。
4. **禁止泛"卡"**。所有反馈都需要落到 [01 量化标准](./01-metrics-standard.md) 的某行指标上。
5. **负责人轮值表每 4 周轮换一次**，禁止同一个人长期透支。

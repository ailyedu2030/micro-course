## 1. 核心原则

### 1.1 角色定义

| 角色 | 职责 | 能力边界 |
|------|------|---------|
| **总工程师（人类）** | 需求决策、架构选型、闸门验收、代码审阅 | 决定做什么、验收是否通过 |
| **AI（opencode）** | 代码实现、测试编写、文档生成 | 执行具体的编码任务 |

### 1.2 决策权分割

```
人类决策              AI 执行
────────────────────────────────────────────
做什么（需求）     →   怎么写（编码）
架构选型           →   代码实现
验收标准           →   让测试通过
代码审阅           →   代码生成
安全/部署审批      →   安全/部署代码初稿
```

### 1.3 八大铁律

```
1. 没有需求 ID 的代码 → 不写      （见第 2 章）
2. 没有数据契约的实体 → 不建表     （见第 3 章）
3. 没有测试的代码 → 不合并          （见第 5 章）
4. 每次 ≤ 200 行 → 超量拆分         （见第 6 章）
5. 安全代码禁止 AI 独立生成          （见第 7 章）
6. 每日审阅 diff → 不超过 15 分钟    （见第 9 章）
7. 每次给 AI 指令必须包含 Negative Prompt（见第 10 章）
8. 每道闸门通过后打 git tag          （见第 11 章）
```

## 3. 数据模型与 API 契约

> 字段不统一、逻辑矛盾、API 不对齐的本质原因：AI 为每个文件独立生成代码，没有统一的"数据字典"做约束。
>
> 解决方案：**契约先行**——每种数据结构只定义一次，所有层引用同一个来源。

### 3.1 双层契约体系

```
┌─────────────────────────────────────────────────┐
│  数据契约（docs/数据字典.md）                      │
│  定义：每个数据库表、每个字段、类型、长度、约束      │
│  用途：后端 Entity ↔ DTO ↔ API  ←→  前端类型      │
├─────────────────────────────────────────────────┤
│  API 契约（docs/API契约.md）                       │
│  定义：每个接口的请求/响应形状（JSON 示例）          │
│  用途：前后端对齐、测试用例生成                     │
│  原则：先定义契约 → 再写实现代码                    │
└─────────────────────────────────────────────────┘
```

### 3.2 数据契约（`docs/数据字典.md`）

#### 3.2.1 命名公约

| 层 | 规范 | 示例 |
|----|------|------|
| 数据库表名 | 小写 snake_case，复数 | `courses`, `users`, `course_chapters` |
| 数据库列名 | 小写 snake_case | `course_id`, `created_at` |
| Java Entity 字段 | camelCase + `@Column(name = "...")` | `courseId` → `course_id` |
| Java DTO 字段 | camelCase | `courseId`, `teacherName` |
| API JSON 字段 | camelCase | `courseId`, `teacherName` |
| TypeScript 类型 | camelCase（与 API JSON 一致） | `courseId`, `teacherName` |
| Vue 变量/Props | camelCase | `courseList`, `currentPage` |

#### 3.2.2 数据契约模板

每个实体按以下格式记录：

```markdown
### 课程 (courses)

| 字段名 | DB 列 | Java 类型 | TS 类型 | 长度 | 约束 | 说明 |
|--------|-------|-----------|---------|------|------|------|
| id | id | Long | number | - | PK, 自增 | 主键 |
| courseId | course_id | Long | number | - | - | 课程 ID（预留外部映射） |
| title | title | String | string | 200 | NOT NULL | 课程标题 |
| coverUrl | cover_url | String | string | 500 | - | 封面图 URL |
| teacherId | teacher_id | Long | number | - | FK→users | 授课教师 ID |
| status | status | Integer | number | 1 | NOT NULL, default 0 | 0=待审核, 1=已发布, 2=下架 |
| createdAt | created_at | LocalDateTime | string | - | NOT NULL | 创建时间 |
| updatedAt | updated_at | LocalDateTime | string | - | NOT NULL | 更新时间 |
```

#### 3.2.3 数据契约的强制规则

```
1. 开发前：先更新数据契约 → 再写建表 SQL → 再写代码
2. 每个实体定义一次，不允许出现"同名不同字段"的歧义
3. AI 如果新增字段或实体，必须先经过你批准再更新契约
4. 同一字段跨层必须名称一致（例如 DTO 的 courseId 在 TS 中也是 courseId）
5. DB 列和 Java 字段名的映射只能通过 @Column 注解完成，不允许手动拼接
```

### 3.3 API 契约（`docs/API契约.md`）

#### 3.3.1 API 响应标准格式

所有 API 响应统一格式：

```json
// 成功
{
  "code": 200,
  "message": "ok",
  "data": { ... }          // 或 data: [ ... ]
}

// 分页
{
  "code": 200,
  "message": "ok",
  "data": {
    "items": [ ... ],
    "page": 0,
    "size": 20,
    "totalElements": 100,
    "totalPages": 5
  }
}

// 失败
{
  "code": 400,
  "message": "参数错误：课程标题不能为空",
  "data": null
}
```

Http 状态码统一用 200，业务状态码通过 `code` 字段区分：

| code | 含义 |
|------|------|
| 200 | 成功 |
| 400 | 参数错误 |
| 401 | 未登录/Token 过期 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 500 | 服务器内部错误 |

#### 3.3.2 API 契约模板

每个 API 按以下格式记录：

```markdown
### GET /api/courses

描述：获取课程列表（分页）
权限：学生/教师/管理员

请求参数：
| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| page | number | 否 | 1 | 页码 |
| pageSize | number | 否 | 20 | 每页条数 |
| categoryId | number | 否 | - | 分类筛选 |
| keyword | string | 否 | - | 搜索关键字 |

响应示例（成功 200）：
{
  "code": 200,
  "message": "success",
  "data": {
    "items": [
      {
        "courseId": 1,
        "title": "高等数学（上）",
        "coverUrl": "https://...",
        "teacherName": "张教授",
        "studentCount": 256,
        "status": 1
      }
    ],
    "total": 50,
    "page": 1,
    "pageSize": 20
  }
}
```

#### 3.3.3 API 契约强制规则

```
1. 开发前：先定义 API 契约（请求+响应形状）→ 你确认 → 再写代码
2. 后端实现必须与契约完全一致（字段名、类型、嵌套结构）
3. 前端按契约开发，不依赖后端"额外返回"的字段
4. 修改 API → 先改契约文档 → 你确认 → 再改实现
5. 同一个数据实体在不同 API 中返回的字段名必须一致
   （例如 /api/courses 和 /api/courses/{id} 都返回 teacherName，不能一个返回 teacherName 另一个返回 teacher）
```

### 3.4 一致性验证（自动化）

#### 3.4.1 提交前检查

每次提交前自动检查：

```
□ API 响应字段名是否与数据契约中的 DTO 字段名一致
□ 同类实体在不同接口中的字段名是否一致
□ DB 列名是否与数据契约中的定义一致
□ Java @Column 注解中的列名是否与数据契约一致
□ 前端 TypeScript 类型是否与 API 契约一致
```

#### 3.4.2 常见的一致性陷阱

| 陷阱 | 错误表现 | 正确做法 |
|------|---------|---------|
| 字段名混用 | 有的接口返回 `courseId`，有的返回 `course_id` | 统一 camelCase |
| 字段冗余 | Entity 和 DTO 混用，接口返回了密码字段 | 严格区分 Entity/DTO |
| 嵌套不一致 | 有的接口 flat，有的 nested | 按 API 契约严格定义 |
| 类型不一致 | Java Long → JS number 精度丢失 | 大数值用 string 传输 |
| 枚举不同名 | 后端 `CourseStatus.PUBLISHED`，前端 `PublishStatus.PUBLISHED` | 前后端枚举名一致 |
| 命名风格漂移 | 开始返回 `createdAt`，后面变成 `createTime` | 同一个字段永远同一个名 |

### 3.5 实际操作：每次开发前的 AI 指令模板

```

- title: String（Java）/ string（TS）
- ...

## 参考 API 契约

## 命名约束

- Java/TS 字段名：camelCase
```

## 4. Phase Gate 闸门制度

### 3.1 四个闸门

```
Gate 1 (Day 7):  用户系统 + 数据库表结构

  - 视频上传 + FFmpeg 转码 HLS
  - 题库管理 + 练习组卷 + 自动批改

  - 学习进度追踪 + 学习打卡
  - 移动端 H5 响应式适配

  - 教务处数据驾驶舱
  - Docker Compose 部署
  - 安全审计 + 性能测试
  - 上线
```

### 3.2 闸门验收标准

每道闸门验收需要同时满足：

```
□ 当前阶段所有功能可手动验证（你走一遍核心流程）
□ 所有测试通过（单元测试 + 集成测试）
□ 不依赖未完成的后续阶段功能
□ 已通过的前阶段功能没有退化
□ 代码中没有超出功能清单的内容
□ 所有 commit 标注了需求 ID
```

### 3.3 闸门验收操作

```bash
# 验收流程
git log --oneline -20          # 检查 commit 是否都有 ID
git diff --stat gate-1..HEAD   # 检查改动了哪些文件
npm run test                   # 跑所有测试
# 手动走一遍核心功能流程
```

**不通过 → 修复或回滚。通过 → 打 tag 进入下一阶段。**

## 5. Test-First 流程

### 4.1 每轮开发的顺序

```
步骤 1: 确定需求 ID（来自功能清单）
步骤 2: 写验收测试（描述接口/行为/边界）
步骤 3: AI 写代码让测试通过
步骤 4: 运行测试 → 通过 → 代码审阅 → 合并
                   不通过 → AI 修复 → 回到步骤 4
```

### 4.2 测试类型分配

| 测试类型 | 比例 | 覆盖范围 | 谁写 |
|---------|------|---------|------|
| 单元测试 | 30% | 工具函数、数据校验、核心算法 | AI |
| 集成测试 | 40% | API 接口、数据库操作、模块间调用 | AI |
| E2E 测试 | 15% | 核心用户流程（选课→看视频→做练习） | AI + 你确认 |
| 静态分析 | 15% | lint、类型检查、安全扫描 | 自动化 |

### 4.3 测试准入标准

```
□ 每个 API 端点至少有 1 个正向测试 + 1 个反向测试
□ 核心算法（如自动批改的分数计算）有边界值测试
□ 数据库操作有 CRUD 基础测试
□ 不降低整体测试覆盖率
```

## 6. 代码增量规则

### 5.1 每次 ≤ 200 行

AI 每次只实现一个需求 ID 对应的功能，生成的代码量不超过 200 行。

```
超过 200 行 → 拆分为子任务：
  例如 S-03 随堂练习拆分为：
  S-03a 练习组卷接口
  S-03b 答题提交接口
  S-03c 自动批改逻辑
```

### 5.2 增量修改原则

```
□ 不要重写已通过的代码
□ 修改既有代码时，用编辑而非重写
□ 每次只改一个模块的文件
□ 不改与当前需求 ID 无关的文件
```

### 5.3 文件创建规则

```
□ 新文件必须与当前需求 ID 直接相关
□ 不要创建工具类/辅助类的"预置"文件
□ 不要创建当前阶段用不到的组件
□ 不要引入新的技术依赖
```

## 7. 安全冻结区

### 6.1 禁止 AI 独立生成的代码

以下代码 AI 可以写初稿，但**必须经过总工程师人工审查后才能合并**：

```
□ 认证/登录（JWT 签发、密码加密）
□ 权限控制（RBAC 逻辑）
□ 视频防盗链签名算法
□ 数据库连接配置
□ Docker/Nginx/HTTPS 部署配置
□ Secrets/密钥管理
□ 数据脱敏逻辑
```

### 6.2 安全代码审查要点

```
□ 密码是否用 bcrypt 加密（不是 md5/sha1）
□ JWT secret 是否从环境变量读取（不是硬编码）
□ SQL 查询是否参数化（不是拼接）
□ 文件上传是否限制类型和大小
□ 视频播放 URL 是否有签名
□ 用户输入是否做了 XSS 防护
□ API 是否有速率限制
```

## 9. 每日审阅流程

### 8.1 你每天的任务（≤ 15 分钟）

```bash
# 1. 查看当天改了哪些文件
git diff --stat HEAD~5..HEAD

# 2. 查看具体的代码改动
git diff HEAD~5..HEAD

# 3. 检查有没有超出范围的内容
#    - 有没有功能清单外的代码？
#    - 有没有不应该出现的文件？
#    - 有没有不该改动的既有文件？
```

### 8.2 审阅重点

```
1. 功能清单外的代码 → 要求 AI 删除
2. 显眼的 Bug → 要求 AI 修复
3. 数据库 schema 不一致 → 回滚
4. 过度抽象/过度设计 → 要求简化
5. 新增不必要的依赖 → 要求移除
```

### 8.3 审阅结果

```
✅ 通过 → 合并
❌ 不通过 → 反馈具体问题 → AI 修复 → 重新审阅
🔄 回滚 → git revert / git reset
```

## 10. Negative Prompt 规则

### 9.1 每轮开发指令必须包含"禁止"部分

```
实现课程列表接口（GET /api/courses）。

要求：
- 返回分页数据
- 支持按分类筛选
- 按创建时间倒序

禁止：
- 不要加分页以外的任何筛选逻辑
- 不要加缓存
- 不要加权限校验
- 不要创建新文件
- 不要修改现有接口
- 不要引入新依赖
```

### 9.2 常见禁止项模板

```markdown
禁止：
- 不要修改与当前任务无关的文件
- 不要创建新的工具类/辅助文件
- 不要加缓存、日志、监控等基础设施
- 不要引入新的第三方依赖
- 不要做代码重构或优化（除非明确要求）
- 不要加认证/权限（除非明确要求）
```

### 9.3 AI 追加了代码怎么办

如果审阅时发现 AI 写了禁止项中的代码：

```
1. 指出具体问题
2. 要求 AI 删除（不是修复，是删除）
3. 如果同一问题重复出现，记录到违规清单
```

## 11. 回滚点制度

### 10.1 Git Tag 规范


### 10.2 什么情况需要回滚

```
□ 数据库 schema 与设计文档严重不一致
□ 新增了大量功能清单外的代码
□ 核心功能在前阶段退化
□ 引入了无法修复的安全漏洞
□ 技术栈偏离了原始设计
```

### 10.3 回滚操作

```bash
# 回滚到上一个稳定状态
git reset --hard v0.1-gate1

# 重新开始
# 注意：使用 git reset 而非 git revert
# 因为我们要彻底放弃偏移的代码
```

## 14. 智能体文件输出规范

### 14.1 问题定义

子智能体（explorer/architect 类型）的文件系统工具仅允许访问 `/Users/jackie/` 路径，而项目目录为 `/Users/jackie/微课平台/`。直接分配文件写入任务给这些类型会导致产物留在 `~/` 而非项目目录。

> **更新历史**：原路径 `/Volumes/Coding/微课平台/` 于 2026-06-11 迁移至 `/Users/jackie/微课平台/`，同步修订 §14.1-14.3。

### 14.2 强制规则

| 规则 | 说明 |
|------|------|
| **文件生成必须用 coder 类型** | 只有 coder 类型有 bash 工具，可通过 `cp` 写入项目目录 |
| **任务末尾附加 cp 步骤** | 每个生成文件的智能体任务，最后一步必须是 `cp <生成文件> /Users/jackie/微课平台/<目标子目录>/<目标文件名>` |
| **总工程师必须确认** | 每次任务完成后，总工程师应验证目标文件已存在于项目目录下的正确位置 |
| **路径一致原则（P5）** | 项目迁移时 §14 必须同步更新；任何"按过时路径 cp"行为 = 偏移 |

### 14.3 标准模板

分配文件生成任务时，prompt 末尾必须包含：

```
## 最后一步
请用 bash 命令将生成的文件 cp 到项目目录：
cp ~/<文件名> /Users/jackie/微课平台/<目标子目录>/<文件名>
```

### 14.4 例外

仅读不写的任务（搜索、分析、审查）可用 explorer 或 architect 类型，不需要 cp 步骤。

| 专家 | 领域 | 核心贡献 |
|------|------|---------|
| AI 开发方法论专家 | 失败模式分析 | 需求-代码追溯、Phase Gate |
| 软件工程过程专家 | 版本控制 | Commit 规范、冻结区间 |
| 测试与质量保障专家 | 测试策略 | Test-First 流程、测试金字塔 |
| 项目管理专家 | 进度风险 | 每日审阅、回滚点制度 |
| 提示工程专家 | Prompt 设计 | Negative Prompt、增量规则 |
| 代码审查与安全专家 | 安全审计 | 安全冻结区、质量门禁 CheckList |

> 详细专家报告见 `docs/专家调研报告.md`

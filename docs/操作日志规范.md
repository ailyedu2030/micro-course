# 操作日志规范 v1.0

> 适用范围：micro-course-api 全部端点与业务 Service
> 关联实现：`com.microcourse.audit.*`、`com.microcourse.service.OperationLogService`、`operation_logs` 表
> 关联任务：P2-10（isAuthenticated 端点统一审计）+ P3-12（操作日志统一接入文档化）

---

## 1. 目的

规范 `operation_logs` 表的写入时机、写入方式、内容格式与性能约束，确保：

- **审计完整**：关键写操作有迹可循（谁、何时、对什么、成功与否）。
- **主流程零退化**：审计是旁路能力，永不阻塞、永不影响用户请求的响应内容与耗时。

---

## 2. 写入时机

### 2.1 必须写日志
- 鉴权操作：登录、登出、修改密码、更新个人信息
- 关键状态变更：课程审核（提交 / 通过 / 驳回）、课程上下架、订单支付、用户状态变更、教学班结课 / 停开
- 资源创建：创建课程、创建选课、创建订单、上传视频
- 批量 / 高权限操作：批量导入用户

### 2.2 可选写日志
- 含敏感数据的读操作（如导出）
- 配置变更

### 2.3 不写日志
- 健康检查、探针
- 公开数据查询（未登录可访问）
- 静态资源访问（封面、视频流等，已有独立访问审计）

---

## 3. 写入方式

本项目提供两种写入入口，**优先使用方式 1（声明式注解）**。

### 方式 1：`@AuditedLog` 注解（推荐）

在 Controller 方法上声明，零侵入业务代码：

```java
@PostMapping("/login")
@AuditedLog("用户登录")
public R<LoginResponse> login(@RequestBody LoginRequest request) { ... }
```

**运行机制（重要）**：审计捕获由 **Spring MVC `HandlerInterceptor`** 完成，而非 AspectJ。

- `com.microcourse.audit.AuditedLogInterceptor`：在 `afterCompletion`（响应已生成之后）读取
  `@AuditedLog` 注解，于**请求线程内**安全取出用户标识、IP、UA、耗时、成功标志，构造
  `OperationLog` 实体，交由 `AuditLogWriter` 异步写入。未标注 `@AuditedLog` 的 handler
  在拦截器首行即透传，**对其余端点零影响**。
- `com.microcourse.audit.AuditLogWriter`：独立 `@Component`，`@Async("taskExecutor")`
  真正异步落库，并对写入异常做 try-catch 隔离。
- 注册位置：`com.microcourse.config.WebMvcConfig#addInterceptors`，作用路径 `/api/**`。

> **为何用 HandlerInterceptor 而非 `@Aspect`？**
> 项目 classpath 仅含 `spring-aop`，**不含 `aspectjweaver`**，且约束「不引入新依赖」。
> `HandlerInterceptor` 来自既有 `spring-webmvc`，零新依赖、零代理开销，与方法级审计场景天然契合
> （实现风格与 `RequireRoleInterceptor` 一致）。

**优点**：声明式、统一、风格一致、自动采集 IP/UA/耗时/状态。
**适用**：Controller 入口级审计（绝大多数场景）。

### 方式 2：直接调用 `OperationLogService`（业务内审计）

当需要在 Service 内部、携带业务上下文（如 `targetId`、自定义 `detail`）记录时使用：

```java
OperationLog entry = new OperationLog();
entry.setUserId(currentUserId);
entry.setAction("USER_STATUS_CHANGE");
entry.setTargetType("User");
entry.setTargetId(targetUserId);
entry.setDetail("status: ACTIVE -> DISABLED");
entry.setSuccess(true);
operationLogService.log(entry);   // 接口签名：void log(OperationLog operationLog)
```

**优点**：灵活，可携带精确的目标对象与业务细节。
**缺点**：易遗漏、风格不统一；调用方需自行保证不阻塞主流程（建议同样放入异步路径）。

> 注意：`OperationLogService.log(...)` 当前为单参（`OperationLog` 实体）签名，**不存在**
> `log(Long, String, String)` 三参重载，调用前请以实体方式构造。

---

## 4. 性能约束（UX 零退化硬约束）

| 约束 | 落地 |
|------|------|
| 异步写入 | `AuditLogWriter.write` 标注 `@Async("taskExecutor")`，`taskExecutor` 由 `AsyncConfig`（`@EnableAsync` + 有界 `ThreadPoolTaskExecutor`）提供，不占用请求线程 |
| 不阻塞主流程 | 拦截器在 `afterCompletion` 阶段 fire-and-forget，不等待写入结果 |
| 失败隔离 | 写入 / 构造异常均被 try-catch 吞掉并仅记 WARN 日志，绝不抛回主流程 |
| 契约不变 | 仅后端内部日志增强，不改变任何 API 响应结构 / 状态码 |
| 线程池有界 | 核心 = CPU×2、最大 32、队列 200、拒绝策略 CallerRuns，防无界增长 OOM |

## 5. 内容格式

`operation_logs` 关键字段（详见 `docs/数据字典.md`）：

| 字段 | 含义 | 注解路径取值 |
|------|------|-------------|
| `user_id` | 操作用户（未登录为 null） | 请求线程内从 SecurityContext 取出 |
| `action` | 操作描述 | `@AuditedLog` 的 value（如 `用户登录`） |
| `target_type` | 目标类型 | Controller 类名（如 `AuthController`） |
| `target_id` | 目标对象 ID | 仅方式 2 业务内填充 |
| `detail` | 细节文本 | `method=Xxx.yyy uri=POST /api/... status=200`（异常时附 `error=...`） |
| `ip` | 客户端 IP | 优先 `X-Forwarded-For` / `X-Real-IP`，回退 `remoteAddr` |
| `user_agent` | UA | 请求头 `User-Agent`（截断 255） |
| `duration_ms` | 处理耗时 | preHandle ~ afterCompletion 差值 |
| `is_success` | 成功标志 | 无异常且 HTTP 状态 < 400 |
| `created_at` | 写入时间 | 落库时填充 |

**`action` 命名约定**：
- 注解 value 使用人类可读中文短语：`用户登录`、`课程审核通过`、`修改用户状态`。
- 业务内枚举式 action 使用 `动词_名词`：`USER_LOGIN`、`COURSE_APPROVE`、`USER_STATUS_CHANGE`。

**长度上限**（拦截器自动截断，防超列宽）：`action` ≤ 100、`detail` ≤ 500、`ip` ≤ 64、`user_agent` ≤ 255。

---

## 6. 数据保留

- 默认保留 **180 天**（业务审计要求）。
- 清理任务建议命名 `CLEAN_OPERATION_LOGS`，可作为定时任务按 `created_at` 滚动清理。

---

## 7. 监控指标（建议）

| 指标 | 目标 |
|------|------|
| 操作日志写入成功率 | ≥ 99% |
| 异步写入对请求耗时的影响 | ≈ 0（旁路，afterCompletion 之后执行） |
| `operation_logs` 表大小 | < 10 GB（配合 180 天保留） |

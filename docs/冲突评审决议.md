## 2. 裁决理由（每条独立说明）

### C1 · REST 路径前缀

**裁决：`/api`（无前缀）**

理由（按权重排序）：
1. **代码现状已用 `/api`**：5 个现有 Controller 全部 `@RequestMapping("/api/...")`，改前缀会破 100% 调用
2. **API契约 v1.1 是 Phase 1 直接契约**：18 个 API 全部用 `/api`（44 处一致）
3. **Phase 1 阶段不需要 v1 标识**：项目未对外发布、无版本兼容性需求
4. **后续若需版本化**：用 `/api/v2` 升级路径而非 URL 前缀

**受影响文档**：权限矩阵 v1.0（141 处 `/api/v1/`）

### C2 · 响应 message

**裁决：`"ok"`**

理由：
1. **API契约 v1.1 是 18 个 API 的直接契约**，54 处一致使用 `"ok"`
2. **开发规范 §3.3.1 是"标准格式示例"**，未指定具体字面值
3. **国际惯例**："ok" 比 "success" 更短、与 HTTP 200 含义一致

**受影响文档**：开发规范 v1.2 §3.3.1

### C3 · 分页字段

**裁决：`items/page/size/totalElements/totalPages`**

理由：
1. **API契约 v1.1 §1398 修订记录**："修复分页格式（统一为 items）"
2. **`totalElements + totalPages` 组合避免前端二次计算** `Math.ceil(total/size)`
3. **Spring Data 生态标准字段名就是 `totalElements`**（Page 接口）
4. **与 18 个 API 的 5 处分页响应示例完全一致**

**受影响文档**：开发规范 v1.2 §3.3.1

### C4 · `/me` 路径

**裁决：`GET /api/auth/me`**

理由：
1. **auth 是身份域、users 是业务域**，`me` 属于"获取当前身份"而非"获取用户业务对象"
2. **与 C1 路径前缀一致**（`/api`）
3. **与 5 个现有 Auth 控制器设计空间一致**（`@RequestMapping("/api/auth")`）
4. **RESTful 语义更清晰**：`/auth/me` = 我的身份；`/users/{id}` = 任意用户

**受影响文档**：权限矩阵 v1.0（3 处 `/users/me`）

### C5 · 表名前缀

**裁决：`users`（无前缀）**

理由：
1. **现有 4 个 Entity 全部无前缀**（`@TableName("users")` 等）
2. **数据字典 v0.4 是 38 张表的权威定义**，全部无 `sys_` 前缀
3. **4 张核心表保持零前缀一致性最高**：`users` / `departments` / `majors` / `classes`
4. **`sys_` 是 Spring 生态历史习惯**，但本项目已不用

**受影响文档**：项目结构规范 v1.0 §4.2

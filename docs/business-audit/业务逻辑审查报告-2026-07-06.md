**异常分支**：
- 登录失败 ≥5 次 → 30 分钟锁定（Redis key `mc:login:lock:{username}`）→ HTTP 423 + code 1006

- 用户被禁用 → JWT 黑名单 + `UserStatusCheckFilter` 30s 内拦截
- 用户被删除 → 软删除（`deleted_at=NOW()`），180 天保留

## 1.3 业务状态机总览

| 状态机 | 状态数 | 后端枚举 | DB 类型 | 前端常量 | 四方一致性 |
|--------|--------|---------|--------|---------|-----------|
| UserStatus | 4 | ✅ Integer code | INTEGER | ❌ 缺失 | ✗ 前端缺失 |
| CourseStatus | 7 | ✅ Integer code | INTEGER | ✅ | ✅ |
| EnrollmentStatus | 7 | ✅ 字符串 | VARCHAR | ✅ + 历史 ENROLLED | ✅（含兼容） |
| OrderStatus | 4 | ✅ 字符串 | VARCHAR | ❌ 缺失 | ✗ 前端缺失 |
| TeachingClassStatus | 3 | ✅ Integer code | INTEGER | ❌ 缺失 | ✗ 前端缺失 |
| MicroSpecialtyStatus | 8 | ✅ Integer code | VARCHAR（不一致） | ❌ 缺失 | ✗ 类型不兼容 |
| MicroSpecialtyEnrollmentStatus | 8 | ✅ Integer code | VARCHAR（不一致） | ❌ 缺失 | ✗ 类型不兼容 |
| MicroSpecialtyFeaturedStatus | 4 | ✅ Integer code | VARCHAR（不一致） | ❌ 缺失 | ✗ 类型不兼容 |
| MicroSpecialtyTeacherRole | 3 | ✅ Integer code | VARCHAR（不一致） | ❌ 缺失 | ✗ 类型不兼容 |
| MicroSpecialtyCompletionRule | 3 | ✅ Integer code | VARCHAR（不一致） | ❌ 缺失 | ✗ 类型不兼容 |
| MicroSpecialtyProposalStatus | — | ❌ 无枚举 | VARCHAR | ❌ 缺失 | ✗ 无枚举 |
| Gender | 2 | MALE/FEMALE | VARCHAR | ✅ | ✗ 缺 SECRET（数据字典允许） |

**结论**：
- 5 个微专业相关枚举（Status/EnrollmentStatus/FeaturedStatus/TeacherRole/CompletionRule）Java 用 Integer code，DB 用 VARCHAR，Entity 只能用 String —— **类型不兼容，无法利用枚举类型安全**
- 申报表 `MicroSpecialtyProposalStatus` 无独立枚举
- 4 个状态机前端常量缺失（UserStatus/OrderStatus/TeachingClassStatus/微专业系列）
- 状态机设计文档 §2.1 `CourseStatus.status=1` 命名 `PENDING`（应统一为 `PENDING_REVIEW`）

### P0-2 · 退课不触发退款（付费用户钱课两空）

| 项目 | 内容 |
|------|------|
| **问题现象** | 付费用户在前端退课后，订单仍为 PAID，不产生退款。设计意图是"退款由 OrderService 统一编排"，但前端退课入口 `MyCourses.vue:822-840` 未引导用户到退款流程，也未自动触发退款 |
| **所属业务流程** | 退课流程（ENR-005）→ 退款流程（ORD-008） |
| **风险等级** | **P0** — 客户必现错误 + 资金风险 |
| **根因定位** | 【后端业务规则】`EnrollmentServiceImpl.cancelEnrollment():473-488` 显式跳过退款（仅打日志）。同时【前端业务规则】`MyCourses.vue:822-840` 仅调退课 API 不调退款 API |

## 4.5 关键洞察（给团队的总结）

### 4.5.1 设计层面
- **微专业状态机是全栈复杂度最高模块**，涉及 8 主状态 + 4 子流程 + DB 触发器 + 跨院审批，建议作为后续重构重点
- **5 个微专业相关枚举与 DB 类型不兼容**（Java Integer code vs DB VARCHAR），是历史设计遗留，需要一次性修复
- **举报流程已 P0**：付费退课钱课两空是当前最严重业务逻辑错误

### 4.5.2 工程层面
- **Service 层校验统一性不足**：部分 Service 用枚举 `canTransitionTo()`，部分用字符串比较，建议制定内部规范统一
- **前端体验降级**主要集中在 `MyCourses`/`Checkout`/`MicroSpecialtyReview` 三个页面
- **后端配置项生效性需逐一验证**：`max_video_size_mb` / `cas_admin_username` 已确认不生效

### 4.5.3 安全层面
- **P0-1（LearningProgress 无选课校验）**是当前最严重漏洞，付费课程内容可绕过支付
- 大量 Service 层调用 `isOwnerOrAdmin()`，但语义不一致（部分允许课主自审批）
- Spring SecurityConfig + `@PreAuthorize` 双重防线设计正确，但需注意 `permitAll` 路径

### 4.5.4 数据层面
- 14 张表关键状态字段无 CHECK 约束，DB 层防御弱
- 状态机四方一致性：DB 列类型与 Java Enum 类型策略不统一
- 软删除覆盖 23 张表（V16 + V80），设计完整

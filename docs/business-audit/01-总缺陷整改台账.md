## 缺陷明细（按严重等级排序）

### P0 级缺陷（必须立即修复）

#### [P0-001] 评价审核流程形同虚设：未审核评价立即可见且影响评分
- **链路**: S20 | **发现者**: Agent 2
- **问题定位**: `CourseReviewServiceImpl.java:126`, `CourseRepository.java:60-64`
- **问题现象**: `listByCourse()` 和 `updateAvgRating()` 的 SQL 未按 `status` 过滤，新创建的评价 `status=0`（待审核）立即可见且参与平均评分计算。审核只是一个空壳。
- **根因判定**: 后端业务规则 — 数据查询层未按审核状态过滤
- **业务影响**: 恶意/垃圾评价可立即展示并操纵评分；审核操作没有实际业务价值
- **修复方案**: 
  1. `listByCourse()` 中追加 `.eq(CourseReview::getStatus, 1)` 仅展示已审核评价
  2. `updateAvgRating()` SQL 追加 `AND status = 1`
  3. 创建评价时默认 `status=0`，更新 avg_rating 在 `approveReview()` 中触发

#### [P0-002] 购物车删除传错 ID（courseId 而非 cartItemId）
- **链路**: S32 | **发现者**: Agent 2
- **问题定位**: `store/cart.js:68-76`, `api/cart.js:16-18`, `CartController.java`
- **问题现象**: `store/cart.js` 的 `removeItem(courseId)` 调用 API 时传入 `courseId` 而非服务端返回的 `cartItem.id`，导致 `DELETE /api/cart/{courseId}` 要么删错项、要么返回 `CART_ITEM_NOT_FOUND`。登录用户无法在购物车中删除商品。
- **根因判定**: 前端交互设计 — 未存储服务端返回的购物车项 ID，删除时错传 courseId
- **业务影响**: 核心功能不可用：购物车无法正常删除商品，每次都报"同步失败"
- **修复方案**: `addItem()` 保存服务端返回的 itemId；`removeItem` 用 `item.id`（cartItemId）而非 `courseId` 调用 API

#### [P0-003] 答题次数检查依赖前端传入值，可绕过
- **链路**: S13 | **发现者**: Agent 3
- **问题定位**: `ExerciseRecordServiceImpl.java:109-114`
- **问题现象**: `submitAnswer()` 检查 `request.getAttemptNo()`，该值来自前端。前端伪造 attemptNo 时可绕过 maxAttempts 限制，产生超出允许次数的答题记录。
- **根因判定**: 后端业务规则 — 检查条件过于宽松（大于而非大于等于）
- **业务影响**: 答题次数限制可被绕过，考试公平性受损
- **修复方案**: 检查条件改为 `request.getAttemptNo() >= exercise.getMaxAttempts()`，或完全不依赖前端值

#### [P0-004] 周报聚合未按周过滤，展示全量累计值
- **链路**: S25 | **发现者**: Agent 4
- **问题定位**: `WeeklyReport.vue:383-411`
- **问题现象**: `calculateReport()` 中 `videoMinutes`/`exerciseCount`/`correctCount` 使用 enrollment 级别累计值而非本周增量。所有统计数据错误地展示了累计值而非周维度数据。
- **根因判定**: 前端交互设计 — 聚合 SQL/逻辑未按周范围过滤
- **业务影响**: 周报完全不可用，展示的是课程总时长而非本周时长
- **修复方案**: 为 `totalWatchTime`, `exerciseCount`, `correctCount` 增加周范围过滤条件

#### [P0-005] 前端驳回原因验证与后端不一致 — CourseApproval.vue
- **链路**: A07 | **发现者**: Agent 10
- **问题定位**: `CourseApproval.vue`, `BatchRejectRequest.java`
- **问题现象**: 后端 `BatchRejectRequest` 要求驳回原因 ≥10 字符，前端 `inputValidator` 只要求 ≥5 字符。前端通过验证但后端报 400 错误。
- **根因判定**: 前端交互设计 — 与后端校验规则不统一
- **业务影响**: 审核人员在批量驳回课程时虽然通过前端校验，但后端拒绝，操作失败
- **修复方案**: CourseApproval.vue 改为 ≥10 字符

#### [P0-006] CourseList.vue handleReject 无输入验证
- **链路**: A07 | **发现者**: Agent 10
- **问题定位**: `CourseList.vue`
- **问题现象**: 使用 `ElMessageBox.prompt` 但不设置 `inputValidator`，驳回原因可为空字符串。
- **根因判定**: 前端交互设计 — 缺少输入校验
- **业务影响**: 课程驳回可无原因，申请人无法知晓驳回理由
- **修复方案**: 添加 `inputValidator`，设置最小长度

#### [P0-007] CourseReviewController 无 approve/reject/delete 端点
- **链路**: A14 | **发现者**: Agent 10
- **问题定位**: `CourseReviewController.java`
- **问题现象**: `CourseReviewController.java` 只有 `create` + `list` 端点，无 `approve`/`reject`/`delete` 端点。前端 `ReviewsManagement.vue` 调用的 API 需确认对应后端实现是否存在。
- **根因判定**: 后端业务规则 — 评价管理的审批/删除端点缺失
- **业务影响**: 管理后台可能无法通过评价管理页面执行审核操作
- **修复方案**: 审计 `review.js` API 文件对应的后端 Controller，确认 approve/reject/delete 端点实现，若缺失则补充

#### [P0-008] 教师评级批量重算可能覆盖手动调整
- **链路**: A22 | **发现者**: Agent 10
- **问题定位**: `TeacherRatingServiceImpl.java`
- **问题现象**: `recalculateAll()` 批量重算可能覆盖管理员的手动调整。需检查 `adjustTier()` 是否设置 `manual_adjustment = true` 标记，`recalculate()` 是否对手动标记的评级跳过。
- **根因判定**: 后端业务规则 — 自动重算未保护手动调整
- **业务影响**: 管理员对教师的评级手动调整会被批量重算覆盖
- **修复方案**: 增加 `manualOverride` 标记和跳过逻辑

#### [P0-009] 用户禁用后选课全链路断裂
- **链路**: A02↔T14↔A13↔A16 | **发现者**: 红队复核(Admin)
- **问题定位**: `UserStatusServiceImpl.java`
- **问题现象**: 用户被管理员禁用(DISABLED)后，已有选课、学习进度、成绩完全不受影响。禁用逻辑仅更新用户状态和清空登录失败计数，未级联处理任何 Enrollment/Progress。
- **根因判定**: 后端业务规则 — 用户禁用未级联学习数据
- **业务影响**: 被禁用学生仍可继续学习、参加考试、获得证书，账号管控形同虚设
- **修复方案**: 
  1. 暂停在学课程（将 enrollment 状态改为 SUSPENDED 或阻断前端入口）
  2. 移除候补队列
  3. 阻止新选课
  4. 学员列表中禁用用户显示"已禁用"标记

#### [P0-010] 评价审核后端 approve/reject API 缺失（待确认）
- **链路**: A14 | **发现者**: Agent 10
- **问题定位**: 后端审核端点
- **问题现象**: 虽然前端 `ReviewsManagement.vue` 有通过/驳回/删除 UI，但 `CourseReviewController.java` 没有对应的 approve/reject/delete 端点。前端调用的 API 可能在另一个 Controller，需审计确认。
- **根因判定**: 需审计确认
- **业务影响**: 管理后台评价审核功能可能不可用
- **修复方案**: 审计 API 文件并补充缺失的端点

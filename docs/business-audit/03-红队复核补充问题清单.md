### R01 — 微专业自动选课静默吞异常

| 项目 | 内容 |
|------|------|
| **涉及链路** | S07 ↔ S29 |
| **遗漏模式** | 异常处理链断裂 |
| **风险等级** | **P1-C** |

#### 问题描述
Agent 5 确认 `MicroSpecialtyEnrollmentServiceImpl.approve()` 在审批通过后循环调用 `enrollmentService.enroll()` 自动注册必修课（第 210-250 行）。但所有异常都被 `catch` 吞掉，仅记入 JSON 字段 `pendingCourses`。如果必修课满员、已下架或价格校验不通过，学生收到"已批准"通知但实际未选上必修课。

#### 根因
- Agent 5 只审查了 `approve()` 的状态流转和乐观锁，未检查异常处理分支
- Agent 2 审查了 `doEnroll()` 的校验逻辑，但不知悉微专业审批流程调用了此方法

#### 修复方案
`approve()` 的异常捕获应区分关键业务异常（满员、课程不可用），关键异常应中断审批流程而非静默吸收；或将 `pendingCourses` 中的失败信息明确返回给前端。

### R02 — 支付后购物车同步必然失败

| 项目 | 内容 |
|------|------|
| **涉及链路** | S32 ↔ S08 |
| **遗漏模式** | 连锁缺陷（同一 Agent 内） |
| **风险等级** | **P1-C** |

#### 问题描述
Agent 2 同时审查了 S32 和 S08 两条链路。S32-7.1 (P0) 发现购物车 `removeItem` 传错 ID 导致服务端删除必然失败；S08-2.1 (P1-C) 发现支付完成后调用 `store.removeItem()`。连锁结果：每次支付完成后用户都看到"同步失败"错误，且服务端购物车记录未被清除。

#### 根因
Agent 2 分别列出了两个独立问题，但未合并为连锁缺陷——S08-2.1 审查支付流程时没有验证 `store.removeItem()` 是否正常工作。

#### 修复方案
修复 S32-7.1（存储服务端 cartItemId）后，S08 支付流程自动恢复正常。

### R04 — 线下签到→进度→统计三级连锁

| 项目 | 内容 |
|------|------|
| **涉及链路** | S12 → S16 → S17 |
| **遗漏模式** | 多跳数据链断裂 |
| **风险等级** | **P1-C** |

#### 问题描述
跨 3 条链路的级联断裂：
- Step 1 (S12)：签到后不更新 `learning_progress`
- Step 2 (S16)：`batchGetByUserAndCourses()` 使用内连接语义过滤掉无进度记录课程
- Step 3 (S17)：`getStats()` 依赖 `learning_progress` 聚合

实际效果：仅含线下课堂的课程，学生签到后课程在所有统计展示中"消失"。

#### 根因
Agent 3 审查 S12 和 S16 但未组合因果链；Agent 4 审查 S17 时假设数据完整。

#### 修复方案
`OfflineSessionServiceImpl.checkin()` 成功后立即调用 `learningProgressService.upsertOrUpdate()`。

### R05 — JVM 时区偏差导致连续打卡徽章误判

| 项目 | 内容 |
|------|------|
| **涉及链路** | S18 ↔ S23 |
| **遗漏模式** | 基础设施→业务逻辑扩散 |
| **风险等级** | **P1-C** |

#### 问题描述
Agent 4 在 S18 发现 `CheckInServiceImpl` 使用 `LocalDate.now()` 依赖 JVM 时区（S18-F6）。同时确认 `checkAndAwardStreak()` 在打卡成功后触发（S23），且读取 `CheckInServiceImpl` 计算的 `streakDays`。连锁：UTC 时区下北京时间午夜前后打卡导致连续天数中断，徽章无法获得。

#### 根因
Agent 4 将 `LocalDate.now()` 视为"打卡模块配置问题"，未上升到影响徽章系统的层面。

#### 修复方案
JVM 参数 `-Duser.timezone=Asia/Shanghai` + 代码层改为 `ZonedDateTime.now(ZoneId.of("Asia/Shanghai")).toLocalDate()`。

### R07 — 通知触发矩阵系统性缺失

| 项目 | 内容 |
|------|------|
| **涉及链路** | S22 ↔ S07/S08/S13/S29 |
| **遗漏模式** | 分片审查制度盲区 |
| **风险等级** | **P1-C** |

#### 问题描述
Agent 4 确认 `NotificationServiceImpl.notifyAsync()` 仅在微专业证书颁发时被调用。各 Agent 本应检查的通知触发点（选课成功、订单变更、成绩发布、讨论回复、审批结果）全部未检查。

#### 根因
通知触发点分布在 N 个 Service 中，Agent 4 只能审查基础设施，无法触达其他 Agent 负责的触发源代码。各 Agent 的 checklist 中无"检查通知触发"条目。

#### 修复方案
按业务场景逐个补充 `notifyAsync()` 调用（选课/支付/成绩/讨论/审批），建议使用 AOP 切面声明式触发。

### R08 — reapply 绕过 RECRUITING 产生僵尸 PENDING 记录

| 项目 | 内容 |
|------|------|
| **涉及链路** | S29 ↔ S07 |
| **遗漏模式** | 审批流向链断裂 |
| **风险等级** | **P1-C** |

#### 问题描述
Agent 5 发现 `reapply()` 缺少 RECRUITING 检查。交叉验证确认 `approve()` 有 RECRUITING 检查。链式结果：学生对已结束微专业 `reapply()` → 创建 PENDING 记录 → 无法 `approve()`（被拦截）→ 学生无法删除（无取消入口）→ 僵尸记录占用表空间、影响重新报名、虚增"已报名"统计。

#### 根因
Agent 5 发现 `reapply()` 缺少检查，但未验证下游 `approve()` 的兼容性，也未分析僵尸记录后续处理。

#### 修复方案
`reapply()` 中补充 `"RECRUITING".equals(ms.getStatus())` 校验。

### Admin R02 — 分类删除后课程 categoryId 成为孤儿引用

| 项目 | 内容 |
|------|------|
| **涉及链路** | A09 ↔ T02 |
| **遗漏模式** | 背景条件崩塌 |
| **风险等级** | **P1-C** |

#### 问题描述
`CourseCategoryServiceImpl.delete()` 仅删分类记录，不处理课程表。如果已有课程关联了该分类，`categoryId` 字段保留旧 ID。课程详情页显示空白/ID，按分类筛选不出现在正确筛选项。

#### 根因
Agent 9 审查分类管理时未检查删除对已有课程的影响；Agent 6 审查课程创建时假设分类数据库永远有效。

#### 修复方案
`delete()` 中检查 `courseRepository.selectCount(eq(Course::getCategoryId, id)) > 0`，有引用时拒绝删除。

### Admin R04 — 成绩锁定触发条件三端各有盲区

| 项目 | 内容 |
|------|------|
| **涉及链路** | T15 ↔ A17 ↔ T16 |
| **遗漏模式** | 责任边界盲区 |
| **风险等级** | **P1-C** |

#### 问题描述
- Agent 7 (T15)：发现成绩无锁定检查，但未说明应根据什么状态锁定
- Agent 10 (A17)：完全未检查成绩修改限制条件
- Agent 8 (T16)：发现教学班结课未锁定成绩，但认为这是成绩模块的责任

核心问题：`grades` 表无 `locked` 字段，课程/教学班 COMPLETED 后成绩仍可修改。

#### 根因
每个 Agent 都认为成绩锁定是对方的责任。

#### 修复方案
以"教学班 COMPLETED"作为成绩锁定触发点，在 `GradeServiceImpl` 的写操作中增加 `Enrollment.status == COMPLETED` 校验。

### Admin R05 — 微专业审批通过后状态为 APPROVED 阻断教师编辑

| 项目 | 内容 |
|------|------|
| **涉及链路** | T19 ↔ AC04 ↔ T20 |
| **遗漏模式** | 上下游断裂 |
| **风险等级** | **P1-C** |

#### 问题描述
`approveAndCreateSpecialty()` 创建 `MicroSpecialty.status = "APPROVED"`（而非 spec 要求的 DRAFT）。教师端 T20 的 `showSubmit` 仅 DRAFT/REJECTED 有操作按钮。教师无法在发布前修改微专业内容，唯一可选操作是直接发布，否则需取消重提。

#### 根因
Agent 8 在 T19 和 T20 分别发现问题但未内部交叉；Agent 11 审查 AC04 时未回溯审批结果在教师端的展示。

#### 修复方案
`approveAndCreateSpecialty()` 将新建状态改为 `DRAFT`，或教师端扩展 APPROVED 状态的可操作范围。

### Admin R06 — 存储申请审批通道断裂

| 项目 | 内容 |
|------|------|
| **涉及链路** | T25 ↔ AC10 |
| **遗漏模式** | 上下游断裂 |
| **风险等级** | **P1-C** |

#### 问题描述
教师端(T25) 存储申请提交功能完整，但教务处(AC10)缺少独立的待审批队列端点。StorageApplicationController 无 `GET /pending` 端点，ACADEMIC 角色无法主动发现有哪些待审批申请。申请可能被无限期搁置。

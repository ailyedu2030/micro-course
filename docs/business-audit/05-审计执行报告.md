### 1.3 审查方法

| 方法 | 说明 | 应用范围 |
|------|------|---------|
| **红队逆向推演** | 从用户视角出发，模拟正常和异常操作路径 | Agent 1-2（学生端主流程） |
| **业务冒烟测试** | 覆盖正反 cases，验证异常分支处理 | 全部 Agent |
| **审计台账** | 按 spec/契约逐项对照检查 | Agent 5（微专业）、Agent 6-8（教师端） |
| **逐文件+跨文件数据流追踪** | 读 Controller → Service → Vue 三层代码，追踪数据流转 | Agent 3-4、Agent 7-11 |
| **知识图谱引用** | 使用 GitNexus 查询函数调用链和代码关系 | 全部 Agent |
| **跨链路红队复核** | 识别分片间的隐性依赖和连锁断裂 | 2 个专用复核 Agent |

### 1.4 分级标准

| 等级 | 定义 | 处理时间要求 |
|:----:|------|:-----------:|
| **P0** | 数据安全 / 核心功能不可用 / 客户首次操作必现错误 | 立即修复（24h 内） |
| **P1-C** | **客户可感知**的不一致、错误消息、体验降级 | 3 个工作日内 |
| **P1-I** | **内部仅见**的代码/文档问题，客户正常使用中不可感知 | 2 周内 |
| **P2** | 代码整洁、安全加固建议、性能优化 | 迭代中逐步处理 |

| 管理端红队 | Agent 6-11 交叉 | 6 组跨链路对 | 6 个（P0: 1, P1-C: 5） |

红队审查方法：对每个预定义的跨链路联动场景，追踪数据从上游 Agent 链路到下游 Agent 链路的完整流转，寻找：
1. 数据流断裂（上游输出 ≠ 下游输入）
2. 异常处理断裂（上游异常未向下游传播）
3. 背景条件崩塌（某个 Agent 的假设被另一个 Agent 的操作破坏）
4. 责任边界盲区（跨模块约束无归属）

合并 11 份分片报告 + 2 份红队报告 → 5 套最终交付物。

## 3. 关键发现模式总结

### 3.1 统计总览

| 指标 | 数值 |
|------|:----:|
| 总缺陷数（去重后） | **162** |
| 其中 P0 | 10（6.2%） |
| 其中 P1-C | 73（45.1%） |
| 其中 P1-I | 56（34.6%） |
| 其中 P2 | 23（14.2%） |
| 红队复核补充问题数 | 14 |
| 红队补充中跨链路遗漏占比 | 8.6% |

### 3.2 六大高频缺陷模式

#### 模式 1：用户体验/交互缺陷（32 个，19.8%）

**典型表现**：
- 按钮可用但后端拒绝（签到全天→分钟级窗口、课程下架仍显示"立即参加"）
- 功能未完成暴露给用户（"回复评价待开发"）
- 操作后无状态反馈（收藏成功/已收藏无区分、批量 AI 静默跳过）
- 异常信息混乱（Token 过期先跳到 Admin Dashboard）

**根因**：前端校验规则与后端不一致、UI 组件未做状态感知的条件渲染、错误处理路径不完整。

#### 模式 2：数据完整性/准确性（25 个，15.4%）

**典型表现**：
- SQL 未按审核状态过滤（评价 `avg_rating` 含未审核数据）
- 统计聚合未按时间范围过滤（周报展示全量累计值）
- 字段映射错误（模板引用不存在的字段名、列绑定不匹配）

**根因**：数据查询层缺少业务条件过滤、跨表聚合未控制维度、VO 字段定义与前端使用不同步。

#### 模式 3：功能缺失/未实现（19 个，11.7%）

**典型表现**：
- 通知触发仅微专业证书一种场景（缺失 7+ 个触发点）
- 证书无自动颁发
- 学生端无收藏列表页面
- 管理端选课列表无审核按钮
- 教务处无存储申请审批 UI

**根因**：后端基础设施已就绪但前端消费侧未覆盖、业务流程的半链路实现。

#### 模式 4：权限控制缺失（17 个，10.5%）

**典型表现**：
- ACADEMIC 角色暴露 ADMIN 的编辑/删除按钮
- `GET /api/users/{id}` 缺少 `hasRole('TEACHER')`
- 匿名帖子 VO 层对教师也隐藏身份
- 成绩页面 ACADEMIC 看到"批改"按钮

**根因**：角色感知的条件渲染缺失、权限矩阵与新角色（ACADEMIC）的适配滞后。

#### 模式 5：并发/竞态安全（8 个，4.9%）

**典型表现**：
- 选课幂等性检查在行锁外、乐观锁缺失（用户状态/申报审批/分账配置）
- 答题次数检查信赖前端值

**根因**：分片审查对并发场景的覆盖不足、版本号乐观锁未统一落实。

#### 模式 6：状态机/枚举不一致（9 个，5.6%）

**典型表现**：
- `REENROLLING` 写入 DB 但枚举未定义
- `resolveStatusLabel()` 将 CANCELLED 映射为"待开课"
- 套餐无状态机校验
- 文档 PROPOSAL_REVIEW vs 代码 PENDING_REVIEW

**根因**：枚举定义与数据写入未同步、状态映射逻辑与枚举定义偏离。

### 3.3 跨链路发现模式

红队复核识别出 **4 种跨链路遗漏模式**，占全部缺陷的 8.6%：

| 模式 | 说明 | 出现次数 |
|------|------|:--------:|
| **上下游断裂** | 上游认为已提交/已审批，下游发现缺失 | 2（Admin R05, R06） |
| **数据流完整性断裂** | 数据从 A→B→C 的链路中在某一步断裂 | 3（Student R03, R04, R06） |
| **异常处理链断裂** | A 的异常被静默处理，B 使用错误的结果 | 2（Student R01, R02） |
| **背景条件崩塌** | Agent 假设某条件永久成立，被另一操作破坏 | 2（Admin R01, R02） |
| **责任边界盲区** | 跨模块约束每个模块都认为责任在对方 | 1（Admin R04） |

**关键结论**：分片审查虽然深度足够，但跨链路数据流和异常传播路径是系统性的盲区。红队复核补充的 14 个问题中，有 **6 个是跨 3 条以上链路的连锁断裂**，这类问题在单链路审查中根本无法发现。

## 4. 改进建议

### 4.1 代码质量改进

**建议 1：建立统一的前后端校验规则管理**
- 将前端校验规则与后端 JSR-303 注解对齐（驳回原因长度、封面大小等）
- 建立 `validation-rules.md` 作为前后端共享的校验规则契约

**建议 2：修复跨模块的 learning_progress 更新断裂**
- 线下签到、练习提交、视频观看均需更新 `learning_progress`
- 建议设计"学习事件总线"统一处理进度更新

**建议 3：补全通知触发体系**
- 选课成功、成绩发布、讨论回复、审批通过等核心场景需补充 `notifyAsync()` 调用
- 建议使用 AOP 切面 + 注解声明式触发

**建议 4：落实乐观锁强制规范**
- 所有状态变更操作必须使用 `version` 乐观锁
- 在代码 review checklist 中增加"状态变更是否有乐观锁"检查项

**建议 5：分账配置、成绩锁定等并发敏感操作增加版本号或锁定机制**

### 4.2 基础设施改进

**建议 6：统一时区配置**
- JVM 参数 `-Duser.timezone=Asia/Shanghai` 在 Docker 镜像中强制设定
- 代码层使用 `ZonedDateTime.now(ZoneId.of("Asia/Shanghai"))` 替代 `LocalDate.now()`

**建议 7：增加配置化管理**
- 审核超时（48h）、预警阈值（30%）、金标上限（2）、置顶上限等硬编码参数迁移到 `application.yml` 或系统设置表

**建议 8：性能优化**
- `getSquareData()` 改为 1 次查询而非 3 次
- `getStats()` 收益查询加时间范围参数
- 批量章节排序改为 CASE WHEN 更新

### 4.3 审计流程改进

**建议 9：审查 checklist 扩展**
在单链路审查模板中加入：
- "后向影响"检查项：我修改/创建的数据，哪些下游模块消费？
- "前向依赖"检查项：我依赖哪些外部条件？（分类存在、用户有效、课程未删除）
- "通知触发"检查项：本链路的核心操作是否需要触发通知？

**建议 10：全生命周期审查**
对于高耦合业务（存储申请、微专业申报、课程审核），审查应强制覆盖"从起点到终点"的完整生命周期，而非只在自己分片内打转。

**建议 11：红队复核制度化**
将跨链路红队复核纳入标准审计流程，至少覆盖：
- 数据流上下游完整性（A 的输出是 B 的输入）
- 异常传播路径（A 的异常如何影响 B）
- 背景条件稳定性（C 的操作会不会破坏 A 和 B 的假设）

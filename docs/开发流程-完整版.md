# 微课平台 · 开发流程完整版

> 本文件由 AGENTS.md 按场景按需加载。
> **编码/提交前必须阅读**，按当前所处阶段对照执行。

---

## 开发流程（强制）

### Step 1 · 读契约
按需读取 contracts（`references/` 下 6 份 + 前端 6 份）确认所有约束。

### Step 2 · 预检
```bash
bash .claude/skills/microcourse/scripts/precheck.sh
```
退出码 != 0 → 禁止写代码。

### Step 3 · 编码
使用 Agent Team 并行开发。用 coder 类型 agent。不用 Lombok、不用 @Autowired 字段注入、构造器注入。

### Step 4 · 交叉验证（强制，不可跳过）
**每次开发阶段完成后，立即启动 5 维交叉验证**：

| 维度 | 审查内容 | Agent |
|------|---------|-------|
| R1 代码质量+契约 | Lombok/@Autowired/分页/响应/@PreAuthorize/ErrorCode | reviewer |
| R2 DB 迁移 | 逐表逐字段 vs 数据字典.md | reviewer |
| R3 安全+配置 | pom.xml CVE / JWT / Redis key / application.yml | reviewer |
| R4 跨域一致性 | FK 链 / 命名 / REST 路径 / Service 接口 | reviewer |
| **R5 前端 UI/UX + 交互逻辑** | **见下方细分 3 路** | **3 reviewer 并行** |

#### R5 前端审查（必查，否则不可上线）

R5 拆为 3 个并行 sub-reviewer，覆盖前端结构性 bug：

| Sub | 焦点 | 必查项 |
|-----|------|--------|
| **R5a 视觉与一致性** | UI 组件规范、主题统一、三态齐全 | Element Plus 组件用法、Loading/Empty/Error 状态、响应式断点、a11y 基础 |
| **R5b 页面与功能交互** | 路由、表单、按钮、用户反馈 | 路由守卫、角色分离、表单提交、token 持久化路径、错误 toast 覆盖 |
| **R5c 数据交互** | Pinia / API / 缓存 / 分页 | store action/getter 一致性、API 调用时机、localStorage/sessionStorage 一致性、分页/列表数据流 |

**铁律**：
- 5 维 reviewer **必须并行**启动，不能串行
- 任一 FAIL → 立即修复 → 重新审查
- 全部 PASS → 才能 git commit

**输出规范**：每条审查结果必须标注 `P0 / P1-C / P1-I / P2`，不标注的审查报告视为无效。

### Step 4.5 · 缺陷修复（禁止表面修复）

交叉验证发现缺陷后，**必须先做根因分析再动手改**：

```
【根因分析】— 修复前必须填写
- 症状：客户看到什么 / 测试报什么
- 直接原因：哪行代码 / 哪个配置（file:line）
- 根本原因：为什么会写错（copy-paste? 规范不清? reviewer 没看? 历史遗留?）
- 类似问题横向扫描：grep/检查同模式是否还有 N 处
- 防止再发：要不要加 precheck.sh 规则 / lint 规则 / 自动化测试
```

**只有根因分析 + 横向扫描完成后才能动手改。** 否则 = 表面修复，必须重做。

修复完成后，重新执行 Step 4 交叉验证（原审查维度 R1-R5），确认：
- P0/P1-C 已清零
- 修复未引入新问题
- 如涉及 P1-I 降级，按发布决策流程申请 defer（详见 `docs/发布管理.md`）

### Step 5 · 创建 Pull Request

所有代码必须通过 PR 合并，**禁止直接推送至 `main` 分支**。

#### 5.1 分支命名规范
```
<type>/<scope>-<描述>
```
类型 | 示例
-----|------
`feature/` | `feature/user-registration-form`
`fix/` | `fix/login-npe-on-null-token`
`chore/` | `chore/upgrade-spring-boot-3.2.12`
`refactor/` | `refactor/extract-pagination-util`
`test/` | `test/add-storage-validator-coverage`
`docs/` | `docs/update-api-contract-phase16`

#### 5.2 分支纪律
- feature/fix 分支从 `main` 拉出，PR 目标为 **`main`**
- 禁止 feature 分支互相合并
- 合并前必须 rebase 到最新 `main`，确保线性历史

#### 5.3 Conventional Commit 类型（完整）
```
类型       | 说明                    | 版本号影响
feat       | 新功能                  | minor
fix        | 修复 bug               | patch
chore      | 构建/工具/依赖变更       | 无
docs       | 文档变更                | 无
refactor   | 重构（不改行为不改接口） | 无
test       | 测试新增或修改           | 无
style      | 代码格式（不影响功能）   | 无
ci         | CI 配置变更             | 无
perf       | 性能优化                | patch
revert     | 回滚                    | 无
```
**非以上类型不得使用**，违者 PR 打回。

#### 5.4 PR 标题与描述模板
```
<type>(<scope>): <描述>

## 变更目的
[一句话说明为什么改]

## 变更内容
- [核心修改 1：file:line]
- [核心修改 2：file:line]

## 关联 Issue
Closes #[issue_number]

## 验证清单
□ precheck.sh 16/16 PASS
□ mvn compile 0 ERROR
□ npm build SUCCESS
□ 单元测试 PASS
□ E2E 测试 PASS
□ 交叉验证通过(R1-R5)
□ P0+P1-C 已清零
```

#### 5.5 签署认证（Sign-off）
每个 commit **必须**包含 Signed-off-by：
```bash
git commit -s -m "fix(scope): 标题"
```

#### 5.6 Code Review 门禁
- **至少 1 名 reviewer 给出 Approved**，方可通过
- Reviewer **不得是 PR 作者本人**（禁止 self-approve）
- 任何 reviewer 给出 "Changes requested" → **阻塞合并**，修复后重新 request review
- Reviewer 必须对照 Step 4 交叉验证结果逐项确认
- Review 通过后 squash merge 入 `main`，**禁止 merge commit，保持线性历史**

#### 5.7 fix 类型必须包含根因分析
```
fix(scope): 标题

【症状】客户看到什么
【根因】为什么会写错（不是改了什么，是为什么之前错）
【横向扫描】grep 结果或类似问题清单
【防止再发】precheck.sh / lint / 测试
【验证】测试命令 + 输出
交叉验证通过(R1-R5) | P0+P1-C 已清零 | P1-I 登记 deferred-items.md
```
**只写"改了什么"的 fix commit = 表面修复，code review 必须打回。**

### Step 6 · 合并与发布

PR 审批通过后，执行 squash merge 入 `main`：

```
git checkout main && git pull --rebase
git merge --squash feature/xxx
git commit -s
```

合并后必须更新：
1. **CHANGELOG.md** — 按 `### Added / Changed / Fixed / Removed / Breaking` 分类记录
2. **Breaking Change** — 任何不兼容变更必须在 CHANGELOG 中以 `BREAKING CHANGE` 标注，并写明迁移指南
3. **Feature Flag** — 大功能默认通过 feature flag 关闭，灰度开启
4. **版本号** — 按语义化版本（SemVer）自动推算


# Hermes 专用调度指令（适配 Hermes 多 Agent 编排，极简强约束、可直接下发调度中枢，贴合 Vibe Coding 半成品项目痛点）

## Hermes 顶层执行指令（最高优先级，Hermes 调度引擎强制遵守，流程不可逆）

任务类型：Vibe Coding 项目全量业务交互专项审计测试

团队上限：最多调度 12 个 Test-Agent 测试智能体

项目固有风险：Vibe Coding 生成代码普遍存在按钮异常、前后端业务逻辑违背行业常识、页面 / 数据交互逻辑错乱、功能半成品残缺、多分支流程一刀切漏洞

### 固定执行流水线（Hermes 不可跳过、不可调换顺序）

1. 启动独立 Parse-Agent（仅做代码解析，不参与测试）
    
    ① 全量拉取前端组件、按钮绑定事件、路由、后端接口、枚举分支、数据库实体代码全域扫描；
    
    ② 提取全部页面、所有功能按钮、操作动作、代码内 if/type 多业务分支；
    
    ③ 最小测试单元定义：【页面 + 单按钮操作 + 单一业务分支】，每条单元附带代码文件溯源路径；
    
    ④ 标记高风险单元：动态显隐按钮、多类型切换流程、新增 / 保存 / 提交接口（Vibe Coding 高发残缺 bug 点位）；
    
    ⑤ 输出《全量测试单元清单》，打散分片分配给 1~12 个 Test-Agent；分配规则：同一链路不同按钮、不同业务分支拆分至不同 Agent，禁止单 Agent 包揽整页面 / 整条链路；生成 Hermes 分片调度分配表。
    
2. 并行调度 12 个 Test-Agent 分片独立红队测试
    
    每个 Agent 仅校验分配到的独立测试单元，禁止合并多按钮、多分支批量粗测；每套单元执行三层校验：
    

#### 校验层 1：按钮基础可用性（解决页面按钮失效异常）

按钮渲染状态、禁用 / 隐藏逻辑、点击事件绑定、重复点击防抖、执行成功 / 失败反馈、切换分支后按钮适配显隐；无响应、白屏、置灰规则错误、多余无效按钮均判定缺陷。

#### 校验层 2：页面 & 数据全链路交互逻辑

路由跳转、弹窗层级、表单回填、跨步骤数据联动、回退 / 中断缓存重置、前后端数据存储一致性；数据错乱、流程断层、刷新丢失数据、多分支数据互相污染记为交互漏洞。

#### 校验层 3：业务常识 & 功能完整性校验（核心抓低级业务错误、半成品功能）

对标行业标准校验分支专属规则、互斥逻辑、业务流程顺序；核查是否存在分支缺失、页面有按钮无后端接口、必填配置缺失、边界场景无兜底；所有违背常规业务认知、半截未完成功能全部标记严重缺陷。

#### Test-Agent 统一输出格式（Hermes 标准化日志）

单元编号｜页面｜按钮名称｜代码路径｜业务分支｜预期业务规则｜实际表现｜缺陷类型（按钮异常 / 交互错误 / 业务常识错误 / 功能残缺）｜风险等级｜根因（前端组件 / 按钮事件 / 后端接口 / 分支判断缺失）｜修复方案

3. Hermes 启动 2 个 Review-Agent 执行双层交叉复核
    
    ① 横向复核：同链路全部分支、全部按钮横向对比，定位一刀切逻辑、部分分支残缺；
    
    ② 纵向复核：完整串联整条业务链路连续操作，挖掘多按钮联动产生的隐性流程冲突；
    
    输出交叉复核补充缺陷台账。
    
4. Hermes 聚合所有 Agent 输出，统一归档交付
    
    汇总 Parse-Agent 清单、各 Test-Agent 单元测试报告、复核缺陷，按四大缺陷类型分区归类：按钮异常、违背业务常识逻辑错误、页面 / 数据交互错乱、半成品残缺功能，输出完整可导出整改总表。
    

## Hermes 强制阻断规则（违反直接终止流程，重启解析分片）

1. 禁止跳过 Parse-Agent 代码扫描环节，不得人工粗略划分测试范围；
2. 禁止以页面、模块、整条链路作为测试单元，必须拆解至「单按钮 + 单业务分支」最小粒度；
3. 禁止同一链路全部按钮 / 全部分支分配给单个 Test-Agent；
4. 禁止模糊化缺陷描述，每条问题必须绑定页面、按钮、代码路径、对应业务分支；
5. 不得忽略 Vibe Coding 典型半成品问题：仅页面渲染无接口、分支区分缺失、互斥业务未做拦截、反常识流程设计。

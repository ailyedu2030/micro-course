## 二、错误巡检机制 (5 个阶段)

### Stage 1: 开发阶段代码评审 (Code Review)
- **触发**: 每个 PR 提交时
- **责任**: 总工程师 + reviewer (项目要求双人审核)
- **工具**: gstack review checklist (SQL Safety / LLM Trust / Naming 等)
- **输出**: PR 评论 (MAJOR/MINOR/COSMETIC 三级)

### Stage 2: 测试阶段功能验证 (Test Verification)
- **触发**: mvn test / npm test
- **责任**: 总工程师
- **覆盖**: 单元测试 18+ 用例 + 集成测试 + E2E
- **SLA**: P0 测试失败立即修复, P1 当日修复, P2 当周

### Stage 3: 预发布环境压力测试 (Staging Load Test)
- **触发**: 每个 Phase 结束 (本次 V309 后)
- **责任**: 总工程师
- **覆盖**: 
  - GET /api/courses/{cid}/courseware/{sid} 100 QPS, p99 < 200ms
  - streamAudio 50 QPS, p99 首字节 < 100ms
  - 并发编辑同 PPT page 触发乐观锁拒绝率

### Stage 4: 生产环境运行监控 (Production Monitor)
- **触发**: 7×24 持续
- **指标**:
  - HTTP 5xx 比例 < 0.1%
  - SQL 慢查询 < 0.1%
  - audio_token 流式 GET 404 率 < 0.5%
  - HtmlSanitizer 触发 XSS 拒绝计数
- **告警**: 任一指标超阈值 → 立即 P0 应急流程

### Stage 5: 用户反馈闭环 (User Feedback Loop)
- **触发**: 用户报告 / 工单 / 反馈
- **责任**: 总工程师
- **流程**: 报告 → 定级 → SLA 修复 → 告知用户 → 复盘

## 三、立即执行流程 (P0 应急响应)

```
0:00  发现 P0 (监控告警 / 用户报告 / 测试失败)
0:05  总工程师确认问题范围 (curl/sql重现 + 容器日志)
0:15  启动修复, 团队通知 (若有 reviewer)
1:00  修复完成, 单元测试 PASS, 提交 + push
1:30  灰度 10% → 监控
2:00  全量 + 用户告知 (微信群 / 邮件 / 状态页)
2:30  根因分析报告 + 知识库更新
```

## 四、根因分析报告模板 (5-Why)

```markdown
# Root Cause Analysis · {P0/P1} · {标题}

## 现象 (What)
- 触发时间 / 影响范围 / 用户感受

## 根因 (Why - 5 层)
1. 直接原因: ...
2. 触发条件: ...
3. 系统缺陷: ...
4. 流程缺陷: ...
5. 设计缺陷: ...

## 修复 (Fix)
- 文件 / 行数 / commit / 测试覆盖

## 防御 (Prevent Recurrence)
- 短期: ...
- 长期: ...
```

## 五、团队复盘机制

- **每旬 (10 天)**: 错误汇总 + 流程优化建议
- **每月**: 错误趋势 + SLA 达成率分析
- **每季度**: 错误模式识别 + 架构级预防

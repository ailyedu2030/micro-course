## 第 1 部分 · 分配规则详解

### 1.1 为什么"按 ID 离散"而不是"按业务域"？

**用户的硬性约束**：
> "审查单元最小颗粒度定义：单条业务链路内的单步用户操作节点，不允许合并多步骤统一审查"
> "禁止单个 Agent 批量负责一整条业务链路、一整个功能模块"
> "分片只分配离散独立操作步骤，不批量分配整条链路、整个模块"

**违反"按业务域分配"的风险**：
- Agent 拿到"学生端 24 个操作单元" → 会形成"学生端视角"，忽略跨域联动问题
- Agent 拿到"微专业 24 个操作单元" → 会聚焦于微专业内部逻辑，忽略微专业与课程/订单/用户的联动

**离散分配的优势**：
- 单个 Agent 必须**跨域思考**，自然避免"局部视角"盲区
- 强制每个 Agent 验证至少 6 个不同业务域的逻辑 → 减少"链路内自洽但跨链断裂"的隐性 bug
- 与阶段 3 的"跨链路红队复核"形成正交验证

### 1.2 round-robin 离散分配公式

```
Agent N 持有 = { OP-K | K ≡ N (mod 12), 1 ≤ K ≤ 318 }
```

即：
- Agent 1：OP-0001, OP-0013, OP-0025, ..., OP-0313（每 12 个取第 1 个）
- Agent 2：OP-0002, OP-0014, OP-0026, ..., OP-0314
- ...
- Agent 12：OP-0012, OP-0024, OP-0036, ..., OP-0318

##### 深度逻辑分析：OrderServiceImpl.batchCreate() 的事务边界问题

`batchCreate()` 方法（OrderServiceImpl:389-404）逐个循环创建+支付：
```java
public List<OrderVO> batchCreate(Long userId, List<Long> courseIds, String paymentMethod) {
    List<OrderVO> orders = new ArrayList<>();
    for (Long courseId : courseIds) {
        OrderVO order = createOrder(userId, courseId, null);
        orders.add(order);
        try {
            order = pay(order.getId(), paymentMethod);
        } catch (Exception e) {
            log.warn("[BatchOrder] 课程 {} 支付失败: {}", courseId, e.getMessage());
        }
    }
    return orders;
}
```

**已发现问题（P1-C 级别）**：
1. **事务不是原子的**：`@Transactional` 注解在方法上，但第一个订单成功后第二个失败会回滚全部。更安全的做法应该是每个订单独立事务。
2. **支付失败不阻断**：失败仅日志记录，前端仍收到成功订单列表。前端依赖订单 VO 列表判断成功，但 pay 异常被 catch 后不往外抛导致前端误判。
3. **已选课检查遗漏**：createOrder 内有已选课检查，但如果部分课程已经在购物车中（之前被购买过），会异常退出循环，前面已支付的订单也被回滚。

**需确认**：前端 `batchCreateOrders` 成功后直接全部标记为成功（Checkout.vue:157-162），实际后端可能部分失败。这是关键不一致点。

##### 深度分析：MicroSpecialtyInviteServiceImpl.acceptInvite()

方法逻辑分解：
1. **身份校验**（:101-104）：必须是本人操作
2. **状态校验**（:107-109）：必须是 INVITED 状态
3. **过期校验**（:112-114）：inviteExpiresAt < now 抛过期异常
4. **终态检查**（:118-120）：微专业已 CANCELLED/ARCHIVED 不可接受
5. **跨学院检测**（:123-128）：比较 offerDepartmentId 与 user.departmentId
6. **分支处理**：
   - **跨院分支**（:130-149）：INVITED → PENDING_ACADEMIC，通知所有 ACADEMIC 用户
   - **同院分支**（:151-172）：INVITED → ACTIVE，若角色为 LEAD 则更新 micro_specialties.lead_teacher_id
7. **通知邀请人**（:174-181）：通知发起邀请的教师

**安全措施**：
- ✅ 乐观锁：`version = version + 1`
- ✅ 状态双重检查：UPDATE 条件包含 `invite_status = 'INVITED'`（P1-1 修复）
- ✅ 跨院通知：P0-2 修复后通知所有 ACADEMIC 而非 userId 0L
- ✅ LEAD 角色双重确认：更新 lead_teacher_id 时也使用乐观锁

**潜在问题（P1-I 级别）**：

1. **并发场景下的过期与状态竞争**（:107-114）：select 后 update 存在间隙。但 UPDATE 条件包含 `invite_status = 'INVITED'` + `version`，即使过期检测通过后并发过期扫描也生效，第二次 UPDATE 会因为 version 不匹配而失败。✅ 安全。

2. **跨院分支的 inviteExpiresAt 未重新检查**：同院分支在过期检查后方进入分支，跨院分支同理。✅ 有检查。

3. **LEAD 角色跨院场景**：若被邀请的是 LEAD 角色且跨院，状态变为 PENDING_ACADEMIC，但 MS 的 lead_teacher_id 不更新（仅在 ACADEMIC 审批时更新）。reviewCrossDept 中对 LEAD 角色更新 lead_teacher_id。✅ 逻辑一致。

4. **前端 MicroSpecialtyList.vue:220-221 的 acceptWithChapters 调用**：还有另一个 `acceptWithChapters` 方法处理带章节决策的接受。`acceptInvite` 是简单接受，`acceptWithChapters` 是附带章节分配的接受。两者均有跨院检测（P1-C-12-06 修复）。

## 重点发现摘要

### P1-C-001：OP-0052 批量结算支付的事务边界不一致

**严重程度**：P1-C — 客户可感知
**位置**：OrderServiceImpl.java:389-404 + Checkout.vue:157-162

**问题描述**：
`batchCreate()` 方法使用 `@Transactional(rollbackFor = Exception.class)` 包裹批量循环，但在循环内 `pay()` 的异常被 catch 吃掉（仅日志），导致：
1. 一批订单中，A 课程支付成功、B 课程支付失败时，事务回滚 B 的同时也回滚了 A
2. 前端在 batch API 成功返回后，将所有课程标记为成功（Checkout.vue:157-162 `successItems.push(...)`），但此时 A 也可能已回滚
3. 降级到逐一处理时，每个订单独立，失败不影响其他，但此时已购入的订单物品已被移出购物车（`store.removeItem`），重试时已不在购物车中

**修复建议**：
方案 A（推荐）：`batchCreate` 内每个订单使用独立事务（`REQUIRES_NEW`），支付失败记录到返回的 VO 中标记失败，前端根据 VO 状态判断而非统一标记成功。
方案 B：降级处理时，在逐个创建订单前先克隆购物车列表，失败后不删除购物车项目，允许用户单独重试。

### P1-I-001：OP-0148 接受邀请——acceptWithChapters 中 updateById 缺乏状态+版本双重CAS

**严重程度**：P1-I — 内部仅见
**位置**：MicroSpecialtyInviteServiceImpl.java:256-262

**问题描述**：
`acceptWithChapters()` 方法中使用 `teacherRepository.updateById(record)` 更新状态，而不是使用 `LambdaUpdateWrapper` 带 `invite_status` 条件 + `version` 的 CAS 更新。虽然 `updateById` 默认带乐观锁 version 校验，但**缺少状态条件**。在与 `acceptInvite()` 并发时可能出现：先被基础接受流程处理了（INVITED→ACTIVE），然后 acceptWithChapters 覆盖更新，导致状态回退到 ACTIVE 被错误修改。

相比之下，`acceptInvite()` 使用 `LambdaUpdateWrapper` + `.eq(inviteStatus, "INVITED")` + `setSql("version = version + 1")` 三重防护，更安全。

**修复建议**：
将 `acceptWithChapters()` 中的 `updateById` 改为带 `invite_status = 'INVITED'` 条件的 `LambdaUpdateWrapper` + 手动 version 递增，与 `acceptInvite()` 一致。

### 重点发现摘要

**1. OP-0003 登录按钮 — 错误码覆盖（P1-C）**
- `AuthServiceImpl.java:80-180` 已实现 IP 级防暴（20 次/15 分钟）和用户级锁定（5 次）
- `ErrorCode.java:9` 定义了 `LOGIN_LOCKED(1006, "登录失败次数过多，账号已锁定", 423)`，返回 423 HTTP 状态
- 前端 `Login.vue` 依赖 `userStore.login()` 的 catch 处理，但未针对 423 做专门的 toast 文案区分——当后端返回 423 时，前端拦截器可能统一处理为 "登录失败" 而非 "账号已锁定，请稍后再试"
- **问题**: `request.js` 中需验证是否对 423 状态码做了差异化处理

**2. OP-0171 批量邀请 — 跨院标记自动分支（P1-C）**
- 前端 `MicroSpecialtyTeamEdit.vue:99` 调用 `inviteTeacher()` 逐个发送邀请请求，未区分跨院/本院
- 后端 `MicroSpecialtyInviteServiceImpl` 的 `acceptInvite()` 和 `acceptWithChapters()` 已实现跨院检测逻辑（`isCrossDept` 变量），但 `invite` 请求本身（创建 `MicroSpecialtyTeacher` 记录）在 `inviteTeacher` 方法中未标记跨院状态
- **风险**: 批量邀请中先失败的教师将导致后续未处理的教师无法被邀请，且前端缺乏"部分成功"的明细展示——前端已实现 `failed` 数组错误上报，但若 `inviteTeacher` 方法自身未做跨院预检，可能导致邀请已发送但后续接受时才发现跨院需要审批

**3. OP-0231 驳回跨院请求 — 理由必填校验（P1-I）**
- 前端 `MicroSpecialtyCrossDeptReview.vue:124-125` 已驳回理由必填校验：`if (!rejectReason.value?.trim())`
- 后端 `reviewCrossDept()` 中 `reason` 参数仅用于通知内容拼接，未做业务校验
- 前后端一致，验证通过

**4. OP-0087 答题超时自动提交 — 当前答案未确认提交（P1-I）**
- `ExerciseTake.vue:865-869` 超时后调用 `doSubmit()`。需确认 `doSubmit()` 是否收集了当前已完成的所有题目答案，还是只提交了已保存的答案。若超时瞬间用户正在答题的未保存答案，可能丢失。

**5. OP-0279 章节保存排序 — 并发风险（P1-I）**
- 前端的 `CourseDetail.vue:202` 上的保存排序按钮调用 `PUT /api/chapters/sort`，处理多个章节的 `sortOrder` 批量更新
- 若两个教师同时编辑排序，后保存的会覆盖先保存的，缺乏乐观锁保护

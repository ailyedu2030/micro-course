## 四、发现的缺陷清单

### BUG-001 [P1-I] — 微专业详情 `permitAll()` SecurityConfig 路径模式不匹配

| 属性 | 值 |
|------|-----|
| **ID** | BUG-001 (M1-03 关联) |
| **严重等级** | **P1-C** |
| **类别** | 权限 / SECURITY |
| **文件** | `micro-course-api/src/main/java/com/microcourse/config/SecurityConfig.java:122` |
| **描述** | SecurityConfig 中 `requestMatchers("GET", "/api/micro-specialties/{id}").permitAll()` 未生效。无认证请求访问 `GET /api/micro-specialties/15` 返回 HTTP 401 "Token 格式错误"，而 `square` 和 `teachers` 端点的 permitAll 正常。 |
| **根因** | `requestMatchers` 创建的 `MvcRequestMatcher` 可能无法正确解析单段 `{id}` 路径模式，导致请求落入 `.anyRequest().authenticated()` 并被 JWT 过滤器拦截。 |
| **影响** | 未登录学生/访客无法查看微专业详情页（课程广场可正常进入，点击详情即跳转登录页） |
| **复现步骤** | `curl http://localhost:8080/api/micro-specialties/15` → HTTP 401 |
| **修复建议** | 将第 122 行改为 `.requestMatchers("GET", "/api/micro-specialties/*").permitAll()`（使用 Ant 通配符 `*` 而非 `{id}`），或同时补 `.requestMatchers("GET", "/api/micro-specialties").permitAll()` 以避免基础路径匹配混乱。 |
| **验证方法** | 修复后 `curl http://localhost:8080/api/micro-specialties/15` 应返回 HTTP 200 + 详情数据 |

### BUG-002 [P1-C] — 学生申请修读 APPROVED 微专业提示"已结束"（错误信息歧义）

| 属性 | 值 |
|------|-----|
| **ID** | BUG-002 (M1-04 关联) |
| **严重等级** | **P1-C** |
| **类别** | UX / 错误消息 |
| **文件** | `MicroSpecialtyEnrollmentServiceImpl.java`（apply 方法） |
| **描述** | 学生申请修读状态为 APPROVED 的微专业时，返回错误码 17013 消息 "微专业已结束，无法操作"。正确的语义应为"微专业当前未在招生期"或"仅招生中的微专业可报名"。 |
| **影响** | 学生看到"已结束"会产生困惑，以为微专业已经关闭，实际微专业仍在 APPROVED 状态等待教师 OPEN。 |
| **复现步骤** | `POST /api/micro-specialty-enrollments/apply` 提交 `{"microSpecialtyId":15}`（MS 15 为 APPROVED 状态） |
| **修复建议** | 在 `apply()` 方法中将 status 检查的逻辑从"不是 RECRUITING = 已结束"分为更精确的判断：如为 DRAFT/PENDING_REVIEW/APPROVED → "微专业当前未开放招生"；如为 COMPLETED/ARCHIVED/CANCELLED → "微专业已结束"。 |

### BUG-005 [P1-C] — `set-gold-featured` 端点返回 500 服务器内部错误

| 属性 | 值 |
|------|-----|
| **ID** | BUG-005 (M3-17 关联) |
| **严重等级** | **P1-C** |
| **类别** | BUG / 异常处理 |
| **文件** | `MicroSpecialtyFeaturedServiceImpl.java:215` + `MicroSpecialtyRepository.java:20` |
| **描述** | `POST /api/micro-specialties/{id}/set-gold-featured` 对所有微专业（包括存在/不存在的 ID）均返回 HTTP 500 "服务器内部错误"。无论是 APPROVED 状态的 MS 15，还是不存在的 MS 99999，均返回 500。这表示异常抛出的不是 `BusinessException`（应由 GlobalExceptionHandler 捕获并返回友好错误），而是一个未被特殊处理的异常。 |
| **影响** | 金标金功能完全不可用——教务处无法设置全校金标微专业 |
| **根因分析** | 最可能的原因是 `msRepository.acquireGoldFeaturedLock()` 中 `@Select("SELECT pg_advisory_xact_lock(42)")` 在 MyBatis 执行时抛出非 `BusinessException` 的异常（如 `MyBatisSystemException` 或事务同步异常），该异常未被 GlobalExceptionHandler 特殊处理，落入 `Exception.class` 兜底 → 500。 |
| **复现步骤** | `curl -X POST http://localhost:8080/api/micro-specialties/15/set-gold-featured -H "Authorization: Bearer $ACADEMIC_TOKEN"` → 500 |
| **修复建议** | 1) 在 GlobalExceptionHandler 中增加 MyBatis/PersistenceException 的友好错误映射；2) 或将 `acquireGoldFeaturedLock()` 替换为行级锁 `SELECT ... FOR UPDATE` + 计数器检查，避免使用 PG advisory lock；3) 在 `setGoldFeatured()` 方法中增加 try-catch 包裹 lock 调用 |

### BUG-006 [P2] — `unset-featured` 状态机守卫偏严

| 属性 | 值 |
|------|-----|
| **ID** | BUG-006 (M3-16 关联) |
| **严重等级** | **P2** |
| **类别** | 状态机 |
| **文件** | `MicroSpecialtyFeaturedServiceImpl.java` |
| **描述** | `POST /.../unset-featured` 仅在 `APPROVED` 状态下允许取消置顶（验证时 MS 15 状态为 APPROVED 但 featuredStatus=NONE，报 17003 "仅已通过状态可取消置顶"）。但"取消置顶"的语义应该允许在任何非终态下将 featuredStatus 重置为 NONE，而不仅仅是 featuredStatus=APPROVED 时。 |
| **影响** | PENDING 或 REJECTED 状态下无法手动清理 featuredStatus（如果有脏数据） |
| **修复建议** | 将守卫条件从 `featuredStatus == APPROVED` 放宽为 `featuredStatus != NONE`，允许在任何非 NONE 状态下取消置顶。 |

### 7.4 补充建议

1. **PG advisory lock 替代方案**：考虑将 `acquireGoldFeaturedLock()` 替换为 `SELECT COUNT(*) ... FOR UPDATE` 行级锁模式，避免 PostgreSQL 版本兼容性问题
2. **SecurityConfig 路径模式审计**：全面审查所有使用 `{pathVariable}` 语法的 `requestMatchers`，替换为 `*` 通配符以确保兼容性
3. **异常处理增强**：在 GlobalExceptionHandler 中增加 `MyBatisSystemException` / `PersistenceException` 的友好处理，防止 500 错误泄漏

### 发现漏洞

#### P1-I: 课程标题/简介/标签缺少 XSS 净化
- **文件**: `micro-course-api/src/main/java/com/microcourse/service/impl/CourseAdminServiceImpl.java`
- **位置**: 第 158、160、161、186 行
- **问题**: `course.setTitle(request.getTitle())`、`course.setSummary(request.getSummary())`、`course.setTags(request.getTags())` 未经过 `XssSanitizer` 净化直接存储

- **修复建议**: 在 `CourseAdminServiceImpl.create()` 和 `update()` 中，对 title 加 `XssSanitizer.sanitizePlainText()`，对 description 已修复，对 summary/tags 加 `XssSanitizer.sanitizePlainText()`

## 3. IDOR（越权）测试

### 测试方法
使用 TEACHER 角色的 token 访问本不应有权限的端点和数据。

### 测试明细

| # | 测试操作 | 角色 | 端点 | 响应码 | 结果 |
|---|---------|:----:|------|:------:|:----:|
| 1 | 查看用户列表 | TEACHER | GET `/api/users` | 200 | ⚠️ 可访问 |
| 2 | 查看其他教师课程 | TEACHER | GET `/api/courses?teacherId=999` | 200 | ⚠️ 可访问 |
| 3 | 删除他人课程 | TEACHER | DELETE `/api/courses/1` | 409 | ⚠️ 业务逻辑阻断（非权限） |

### 代码审计结果

| Controller | 端点 | @PreAuthorize | 是否越权 |
|-----------|------|---------------|:--------:|
| UserController | GET `/api/users` | `TEACHER/ACADEMIC/ADMIN` | ⚠️ Service 层限制 TEACHER 仅看自己课程学生（第 92-113 行），但 Controller 级别权限宽于最小原则 |
| CourseController | DELETE `/{id}` | `TEACHER/ADMIN` | ⚠️ Controller 层不做 owner 校验，依赖 Service 层 `isOwnerOrAdmin` |

### 发现漏洞

#### P1-C: TEACHER 可通过 `/api/users` 列表查看全校用户
- **文件**: `micro-course-api/src/main/java/com/microcourse/controller/UserController.java`
- **位置**: 第 47 行
- **问题**: `@PreAuthorize("hasAnyRole('ADMIN', 'ACADEMIC', 'TEACHER')")` 允许 TEACHER 访问全局用户列表
- **缓解**: Service 层 `UserServiceImpl.pageUsers()`（第 92 行）对 TEACHER 角色做了数据范围限制，仅返回其课程内的学生。但该限制仅在 `keyword=null` 且 `role=null` 时生效；若传入 `keyword=admin` 或 `role=ADMIN`，可能绕过范围限制
- **修复建议**: 收紧 Controller 权限至 `ADMIN,ACADEMIC`，或确保 Service 层查询条件在所有参数组合下都能正确限制范围

#### P1-C: TEACHER 可查看全校课程列表
- **文件**: `micro-course-api/src/main/java/com/microcourse/controller/CourseController.java`
- **位置**: 第 61 行
- **问题**: `@PreAuthorize("isAuthenticated()")` 任何已登录用户均可查看课程列表（含其它教师课程）
- **缓解**: 课程广场设计上所有课程为公开可浏览，但 `teacherId=999` 应返回空或 404（实际返回 200+空列表，是数据查询结果而非权限阻断）
- **修复建议**: 若业务要求课程仅本教师/管理员可见，需在 Service 层添加权限过滤

## 6. 请求频率限制测试

### 测试方法
对登录和 API 端点发起高速重复请求，验证限流/锁定机制。

### 测试明细

| # | 测试操作 | 结果 | 阈值 |
|---|---------|:----:|:----:|
| 1 | 50 次错误密码登录 | ✅ 第 6 次触发锁定（1006/423） | 5 次失败 |
| 2 | 20 次高速调用 GET `/api/courses` | ❌ 无限流 | — |
| 3 | 封面文件限流 | ✅ 有 `FileAccessRateLimitInterceptor` | 60 req/min/IP |

### 代码审计

| 限流点 | 实现方式 | 状态 |
|--------|---------|:----:|
| 登录失败锁定 | Redis 计数器，5 次失败锁定 | ✅ |
| Refresh Token 限流 | Redis 计数器，20 次/小时/IP | ✅ |
| 封面文件枚举 | `FileAccessRateLimitInterceptor`，60 次/分钟/IP | ✅ |
| 申请表 autoSave | JVM 内存限流，1 秒/次 | ✅ |
| **通用 API 限流** | **不存在** | **❌ P2** |

### 发现漏洞

#### P2: 缺少通用 API 限流
- **问题**: 登录、refresh、封面文件之外的所有 API 端点（课程列表、用户列表等）没有请求频率限制
- **影响**: 攻击者可高速遍历 `/api/courses`、`/api/users` 等端点进行数据爬取或简单 DoS
- **修复建议**: 
  - 使用 Spring AOP + Redis 实现全局 API 限流中间件（如 `@RateLimit` 注解）
  - 或在网关层（nginx）配置 `limit_req` 模块

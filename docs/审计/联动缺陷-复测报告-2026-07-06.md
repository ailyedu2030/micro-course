## 6.1 LD-002: 进度永久丢失复测（原 P0）

### 缺陷回顾
`VideoPlayer.vue` 的 `reportProgress()` 在 API 调用前就更新了 `lastReportedProgress`（L990-991），API 失败后 catch 块不重置。下次上报因阈值过滤（`progressPercentVal === lastReportedProgress`）跳过，导致该时间点的进度永久丢失。

### 复测方法
白盒代码审查 `micro-course-admin/src/views/student/VideoPlayer.vue`。

### 复测结果

**文件**: `micro-course-admin/src/views/student/VideoPlayer.vue`  
**行号**: L1005-1012  
**状态**: ✅ **已修复**

**修复证据**:
```javascript
// L1005: API 成功后再更新 lastReportedProgress
await updateLearningProgress(progressId.value, { ... })
lastReportedProgress = progressPercentVal      // L1006：API 成功后更新
lastFailedProgress = null                       // L1007：清除失败标记

// L1011: API 失败时记录失败值，确保下次能重试
catch (e) {
  lastFailedProgress = progressPercentVal        // L1011：记录失败值
}
```

**修复详情**:
1. **`lastReportedProgress` 移至 API 成功后更新**（L1006）— 原 bug 是在 L990-991 的 API 调用前更新，现在只有 `updateLearningProgress()` 成功后才赋值
2. **失败标记 `lastFailedProgress`**（L1011）— catch 块记录本次失败的进度值，配合守卫条件确保定时器下一轮（10s）能**自动重试**
3. **成功时清除失败标记**（L1007）— `lastFailedProgress = null`，恢复正常阈值过滤
4. **指数退避重试**：第二次失败前等待 3s，第三次等待 5s

**结论**: 原缺陷的根因（先赋值后调用 + catch 不重置）已完全消除，且引入了失败重试机制。✅

```java
public List<OrderVO> batchCreate(Long userId, List<Long> courseIds, String paymentMethod) {
    List<OrderVO> orders = new ArrayList<>();
    for (Long courseId : courseIds) {
        OrderVO order = createOrder(userId, courseId, null);
        try {
            orders.add(pay(order.getId(), paymentMethod));
```

## 6.5 LD-011: 计数器共享冲突复测（原 P0）

### 缺陷回顾
`AuthServiceImpl` 中登录 IP 防暴和 refresh Token IP 限流使用相同 Redis key 模式 `login_failure:{clientIp}`。暴力登录触发 20 次上限后，合法用户的 refresh 请求也被同一计数器拦截。

### 复测方法
白盒代码审查 + API 级验证。

### 复测结果

**文件**: `micro-course-api/src/main/java/com/microcourse/service/impl/AuthServiceImpl.java`  
**行号**: L160-166（登录防暴）、L259-268（refresh 限流）  
**辅助文件**: `RedisUtil.java`  
**状态**: ✅ **已修复**

**修复证据 — Redis key 已分离**:

```java
// AuthServiceImpl.java L259-266 (refresh 限流)
// Step 0: IP 级别防刷新 — 同一 IP 每小时最多刷新 20 次
// (P0-L02 修复: 独立 key, 不与登录失败计数共享)
String clientIp = IpUtil.getClientIp();
if (clientIp != null) {
    int refreshCount = queryService.getRefreshCount(clientIp);
    if (refreshCount >= 20) {
        throw new BusinessException(ErrorCode.TOKEN_INVALID);
    }
}
```

| 功能 | Redis Key 模式 | 隔离状态 |
|------|---------------|:--------:|
| 登录 IP 防暴 | `mc:login:lock:ip:{clientIp}` | ✅ 隔离 |
| Refresh 限流 | `mc:refresh:limit:{clientIp}` | ✅ 隔离 |

### API 级验证

```bash
    -d '{"username":"student","password":"wrong"}'
done
# 第5次: code=1001; 第6次: code=1006 账号已锁定

# Step 2: 登录锁定状态下 refresh token 正常工作

  -H "Content-Type: application/json" \

```

**修复详情**:
1. **新增独立 Redis key 前缀** `mc:refresh:limit:` — 与登录失败计数的 `mc:login:lock:` 完全隔离
2. **新增独立 Service 方法** `getRefreshCount()` / `incrRefreshCount()` 等
3. **本地降级缓存也独立**：`AuthQueryServiceImpl` 中 refresh 回退使用 `"refresh:" + clientIp` 作为 cache key

**结论**: 两个计数器在 Redis 层面和本地缓存层面均不共享 key，合法用户的 refresh 请求不会被登录防暴错误计数拦截。✅

## 2. 审计方法论

采用 **4 阶段红队审计** 方法论：

### Phase 1 — 横向拆分（12 Agent 并行白盒测试）
将 50 个功能点拆解为 104 个原子测试单元（TC-001 ~ TC-104），分配给 12 个并行 Agent。每 Agent 负责 3~14 个测试单元，覆盖：
- **功能验证**: 按"前置条件→测试步骤→预期结果→API 验证"四步执行
- **状态机验证**: 5 种状态 × 6 种操作的完整矩阵 = 30 个检查点
- **并发验证**: 乐观锁冲突、autoSave 限流、读写事务隔离
- **安全验证**: 路径穿越防御、魔数校验、IDOR 防护、自审批阻断
- **边界验证**: 2MB 文件上限、超长标题截断、特殊字符过滤
- **通知验证**: 审批/驳回通知生成确认

### Phase 2 — 横向交叉比对（Horizontal Review）
对比 5 个横切维度在各状态下的行为一致性：
1. 重置功能在不同模块的表现
2. 5 种状态的按钮矩阵前后端一致性
3. 提交校验的分支差异
4. autoSave 在各状态下的行为一致性
5. 导出功能在 5 种状态下的一致性

### Phase 3 — 纵向全流程串联（Vertical Review）
执行 5 条完整端到端业务链路，验证数据在跨步骤/跨状态流转中的一致性和状态机完整性：
1. 教师端完整旅程（创建→填写→提交→审批→结果）
2. 驳回→重提→再审批闭环
3. 撤回→删除全链路
4. autoSave + PUT 竞争条件
5. ACADEMIC 审批端完整流程

### Phase 4 — 全量缺陷归集与根因定位
合并 12 Agent 发现的全部缺陷，横向交叉复测结果，纵向端到端问题，统一按 P0/P1-C/P1-I/P2 分级归集，每缺陷精确到 `文件:行号`。

### P1-C 级 — 客户可感知的缺陷（10 个，其中 4 个已修复）

#### B-002: SignatureBlock props 不匹配，上传函数静默丢失

#### B-005: `initDraft()` 未创建 3 行固定签字

#### B-006: `created_micro_specialty_id` backlink 缺失
- **等级**: P1-C
- **位置**: `MicroSpecialtyProposalServiceImpl.java:475-476`
- **描述**: 批准申报后，`approveAndCreateSpecialty()` 先通过 `LambdaUpdateWrapper` 更新 `status=APPROVED`（`version` 自增 +1），接着用同一 `proposal` 对象（`version` 仍为旧值）调 `updateById` 写回 `createdMicroSpecialtyId`。因 `@Version` 乐观锁使 `WHERE version=0` 不匹配 DB 中的 `version=1`，导致 0 行更新静默失败。所有通过此路径审批的 proposal 均丢失 backlink，micro_specialty 记录变为"孤儿"。
- **根因**: `approveAndCreateSpecialty()` 未在 update 前重新 `selectById` 刷新 version（对比 `approveProposal()` 第 194 行正确执行了刷新）
- **状态**: ✅ **已修复** — 在第 474-476 行前补充 `proposal = proposalRepository.selectById(proposalId)`
- **修复验证**: 批准后 DB 中 `created_micro_specialty_id` 已正确填充

#### B-007: 驳回原因不显示（`reviewComment` 缺失）

#### B-008: `parseDate()` 不支持 `yyyy-MM-dd` 纯日期格式
- **等级**: P1-C
- **位置**: `StorageApplicationCudServiceImpl.java` → `parseDate()` 方法
- **描述**: `parseDate()` 仅支持 `yyyy.M`、`yyyy.M.d`、`yyyy-MM`、ISO 日期时间格式。纯 `yyyy-MM-dd`（如 `2026-09-01`）匹配不到任何分支，进入 ISO 分支后 `LocalDateTime.parse("2026-09-01")` 抛异常 → `catch` 返回 null。结果：用户输入的 `startDate` 静默丢失写为 NULL。
- **根因**: `parseDate()` 缺少 `yyyy-MM-dd` 格式分支
- **状态**: ✅ **已修复** — 添加 `yyyy-MM-dd` → `LocalDate.parse(trimmed).atStartOfDay()` 分支
- **修复验证**: 传入 `"startDate": "2026-09-01"` 后 DB 正确存储

- **状态**: ✅ **已修复** — 4 个端点（GET /{id}, GET /preview, export-word, export-pdf）的 `@PreAuthorize` 已补充 `'ADMIN'`
- **修复验证**: ADMIN → `GET /{id}`, `GET /preview`, `export-word`, `export-pdf` 全部 200 ✅

#### H-01: WITHDRAWN 导出未被后端拦截

### P1-I 级 — 内部仅见代码/文档问题（9 个）

#### I-01: JSON `message` 含字面换行符
- **等级**: P1-I
- **位置**: `StorageValidator.java:104-105`
- **描述**: `validateAndThrow()` 使用 `String.join("\n", errors)` 拼接错误消息，该消息直接放入 `BusinessException.message`。序列化后 JSON 字符串含未转义的 control character `0x0a`，违反 RFC 8259 §7，导致标准 JSON 解析器解析失败。
- **状态**: ✅ **已修复** — `StorageValidator.java:104-106` 改用 `String.join("; ", errors)` 以 `; ` 分隔拼接，消除原始 `\n` 字面换行符
- **修复验证**: 提交不完整表单 → 返回 JSON message 不含未转义换行符

#### I-02: PNG 伪装 `.jpg` 魔数校验可通过
- **等级**: P1-I
- **位置**: `StorageApplicationServiceImpl.java:285-293`
- **描述**: 魔数校验仅验证内容是否为**任意**支持格式（JPEG 或 PNG），未做扩展名-内容交叉校验。PNG 魔数匹配后 `validMagic = true`，绕过了校验。PNG 内容伪装为 `.jpg` 扩展名的文件可通过。
- **状态**: ✅ **已修复** — `StorageApplicationServiceImpl.java:293-330` 将 magic 校验拆分为 `isJpegMagic` / `isPngMagic` 两个独立标记，增加扩展名-内容交叉验证：JPEG 魔数需 `.jpg/.jpeg` 扩展名，PNG 魔数需 `.png` 扩展名，不匹配则抛 `SA_SIGNATURE_IMAGE_INVALID_TYPE`
- **修复验证**: 上传 PNG 内容伪装 `.jpg` 扩展名 → 400 "文件内容与扩展名不匹配"

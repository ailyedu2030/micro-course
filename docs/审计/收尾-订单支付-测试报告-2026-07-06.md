## 1. 资金安全（4 项核心检查）

### 1.1 S-01: 重复支付 → 幂等性 ✅

**验证路径**: `POST /api/orders/{id}/pay`

**防御层次**:
1. **状态机白名单**: `OrderStatus.canTransitionTo(PAID)` 仅允许 `PENDING → PAID`
2. **@Version 乐观锁**: `updateById()` 自动拼接 `WHERE version = ?`，`affected == 0` 回滚
3. DB 唯一索引: `uk_orders_user_course_active` 防止同一用户同一课程的第二个活跃订单

**测试结果**:
```
1. createOrder(courseId=5, bundleId=7) → PAID (free)  ✓
2. POST /orders/1/pay → 8004 "订单当前状态不允许支付"  ✓ (REFUNDED can't pay)
3. POST /orders/1/pay (again) → 8004  ✓
```

### 1.2 S-02: 退款后再次退款 → 拦截 ✅

**验证路径**: `POST /api/orders/{id}/refund`

**防御层次**:
1. **状态机白名单**: `canTransitionTo(REFUNDED)` 仅允许 `PAID → REFUNDED`
2. **内存守卫**: `REFUNDING_ORDERS` ConcurrentHashMap 防止重入
3. **@Version 乐观锁**: 并发 CAS 保护

**测试结果**:
```
1. POST /orders/1/refund → 200, status=REFUNDED  ✓
2. POST /orders/1/refund → 8004 "订单当前状态不可退款"  ✓
3. POST /orders/1/pay → 8004 "订单当前状态不允许支付"  ✓
4. POST /orders/1/cancel → 8004 "订单当前状态不可取消"  ✓
```

### 1.3 S-03: 非本人订单查看 (IDOR) ✅

**验证路径**: 所有涉及订单/证书的端点和操作

| 端点 | 防御机制 | 测试结果 |
|------|---------|---------|
| `GET /api/orders/{id}` | `isOwnerOrAdmin()` | IDOR user → 10003 "无权限操作" ✅ |
| `POST /api/orders/{id}/pay` | `isOwnerOrAdmin()` | IDOR user → 10003 ✅ |
| `POST /api/orders/{id}/cancel` | `isOwnerOrAdmin()` | IDOR user → 10003 ✅ |
| `POST /api/orders/{id}/refund` | `isOwnerOrAdmin()` | IDOR user → 10003 ✅ |
| `GET /api/orders/my` | JWT userId 硬绑定 | IDOR user 只能看到自己的 ✅ |
| `GET /api/cart` | JWT userId 硬绑定 | IDOR user 看到自己的空购物车 ✅ |
| `DELETE /api/cart/{itemId}` | `item.userId == currentUserId` | 非本人 item → CART_ITEM_NOT_FOUND ✅ |
| `GET /api/certificates/{id}` | `getByIdWithOwnerCheck()` | 代码审计通过 ✅ |
| `GET /api/certificates/{id}/download` | `downloadCertificateWithOwnerCheck()` | 代码审计通过 ✅ |
| `POST /api/certificates/issue` | ADMIN only | Student → 403 ✅ |

**结论**: 所有关键操作的 IDOR 防护到位。使用 `SecurityUtil.isOwnerOrAdmin()` 统一方法，无遗漏。

### 1.4 S-04: 金额篡改 → 不可篡改 ✅

**验证结论**: 价格在服务端从数据库加载，客户端无法传入金额参数。

| 请求体 DTO | 包含的字段 | 金额来源 |
|-----------|-----------|---------|
| `OrderCreateRequest` | courseId, bundleId | 服务端查课程/套餐价格 |
| `BatchOrderRequest` | courseIds, paymentMethod | 服务端逐课程算价 |
| `PayRequest` | paymentMethod | 不涉及金额 |

**DB 级防御**:
```sql
-- V165: CHECK 约束防止负数金额
ALTER TABLE orders ADD CONSTRAINT chk_orders_amount CHECK (amount >= 0);
```

## 2. 订单状态机验证

### 2.1 状态流转白名单（OrderStatus.java）

```
PENDING  → PAID        (支付)
PENDING  → CANCELLED   (取消/超时)
PAID     → REFUNDED    (退款)
CANCELLED → (终态)      ❌ 任何流转
REFUNDED  → (终态)      ❌ 任何流转
```

### 2.2 实测验证

| 测试场景 | 结果 |
|---------|------|
| PAID 订单尝试取消 | ❌ 8004 "订单当前状态不可取消" |
| PAID 订单退款 | ✅ REFUNDED |
| REFUNDED 订单再退款 | ❌ 8004 |
| REFUNDED 订单再支付 | ❌ 8004 |
| REFUNDED 订单再取消 | ❌ 8004 |

### 2.3 并发安全（@Version 乐观锁）

`Order.java` 的 `version` 字段结合 `@Version` 注解（V65 migration），所有状态变更均通过 `updateById()` 实现 CAS：

```java
int affected = orderRepository.updateById(order); // 自动 WHERE version = ?
if (affected == 0) { throw new BusinessException(...); }
```

DB 侧唯一索引兜底：
```sql
CREATE UNIQUE INDEX uk_orders_user_course_active ON orders (user_id, course_id) WHERE status IN ('PENDING', 'PAID');
```

## 3. 发现的问题

### 3.1 B-01: 免费课程下单时 autoEnroll 时序缺陷 [P1-I]

**风险等级**: P1-I（内部仅见，客户无法触发）

**发现路径**: `POST /api/orders` with courseId=4 (free course, not enrolled, selectable)

**实际现象**:
```
POST /api/orders {"courseId":4} → 9005 "非法支付来源"
```

**根因分析**:

在 `OrderServiceImpl.createOrder()` 中，免费课程走以下路径：

```
free order path:
  → 计算价格: isFreeOrder = true
  → autoEnroll(userId, courseId)     ← ⚠️ 先选课
    → enrollmentService.enroll(req) with sourceChannel="PAYMENT"
      → EnrollmentServiceImpl: 检查是否有 PENDING/PAID 订单？没有 → throw "非法支付来源"
  → (永远到不了这步)
  → 创建 order 并 insert
```

**调用链**:
- `OrderServiceImpl.createOrder(L91)` → L172 `autoEnroll()` → L557 `enrollmentService.enroll()` → `EnrollmentServiceImpl.enroll(L178)` → 检查 PAYMENT sourceChannel 需有 PENDING/PAID 订单 → 订单未创建 → 抛出 `9005 "非法支付来源"`

**影响范围**:
- 所有免费（price=null/isFree=true）且未被定价的课程
- 不影响付费课程（付费课程流程走 `isFreeOrder=false`，不调用 `autoEnroll`）
- 不影响套餐下单（套餐调用 `enrollBundleCourses()` 使用 `BUNDLE` sourceChannel，不触发 PAYMENT 检查）

**修复建议**:
```java
// 方案 A（推荐）：调换 order creation 和 enrollment 的顺序
if (isFreeOrder) {
    // 1. 先创建订单
    Order freeOrder = new Order();
    freeOrder.setAmount(BigDecimal.ZERO);
    freeOrder.setStatus("PAID");
    // ...
    orderRepository.insert(freeOrder);
    
    // 2. 再选课（此时已有 PAID 订单，可通过 sourceChannel=PAYMENT 验证）
    autoEnroll(userId, courseId);
    return toVO(freeOrder);
}

// 方案 B：对 free order 使用非 PAYMENT sourceChannel
// 在 autoEnroll 中判断若订单为 free，传 sourceChannel="FREE"
```

## 5. 安全架构评估

### 5.1 防御层级

```
Spring Security (未认证→401)
  └── @PreAuthorize (角色白名单)
       └── SecurityUtil.isOwnerOrAdmin() (行级 IDOR 防护)
            └── OrderStatus.canTransitionTo() (状态机白名单)
                 └── @Version 乐观锁 (并发 CAS)
                      └── DB 约束 (CHECK + 部分唯一索引)
```

### 5.2 关键发现总结

| # | 类别 | 发现 | 风险 |
|---|------|------|------|
| 1 | ✅ 资金安全 | 4 项核心检查全部通过 | — |
| 2 | ✅ 状态机 | PENDING→PAID→REFUNDED 路径完整 | — |
| 3 | ✅ IDOR | 所有端点均有 owner/role 校验 | — |
| 4 | ✅ 金额校验 | 服务端定价，请求体不传金额 | — |
| 5 | ✅ 并发安全 | @Version + 部分唯一索引 + CHECK 约束 | — |
| 6 | ⚠️ 回调验证 | dev 模式无条件放行，prod 强制 | 需确保 prod 配置 PAY_CALLBACK_SECRET |
| 7 | ❌ 免费课程缺陷 | autoEnroll 时序错误导致无法下单 | P1-I（内部仅见，无客户数据） |

### 5.3 改进建议

1. **P1-I 修复**: 修复 `OrderServiceImpl.createOrder()` 中免费课程 autoEnroll 与订单创建的时序问题（详见 §3.1）
2. **P2 改进**: 购物车添加不存在的课程时，返回清晰错误（当前返回 409 "数据冲突"，期望返回课程不存在）
3. **生产配置提醒**: 确保 `payment.callback-secret` 在生产环境通过环境变量注入，`payment.mode` 设为 `real`
4. **日志增强**: 回调验证失败时记录更多上下文（IP、时间、请求体）以便事后追溯

## 一、Agent Team 结构

```
[总工程师 = 主 Agent,审判员]
        │
        ├─ red   (安全红队, 60分钟)
        │     └─ IDOR / SQLi / XSS / 越权 / 暴力破解
        │
        ├─ perf  (性能, 60分钟)
        │     └─ 并发压测 / 连接池 / 响应时间
        │
        ├─ ux    (客户体验, 90分钟)
        │     └─ 33 条真实路径 / 134 张截图
        │
        ├─ ops   (基础设施, 30分钟)
        │     └─ 8 项部署条件逐项验证
        │
        └─ review (交叉验证, 30分钟)
              └─ 复核 4 份 agent 报告,降级/剔除误报
```

**核心原则**: Agent 必须 **真的跑** 真的测,不是描述.每个发现必须有:
- 真实 URL/命令
- 真实响应/截图
- 复现步骤

## 五、Agent Team SOP 总结

### 5.1 经验教训

1. **必须交叉验证**: red/ux 误报率 60%+,没有 review 会被严重误导
2. **必须真的跑**: agent 容易 "描述" 而非 "执行",需要明确要求证据
3. **必须有边界**: agent 不应深入非授权范围,reviewer 限制 scope creep
4. **主 Agent 必审**: 5 项亲自抽查,抓到 2 个误报,证明 agent 不能全信

### 5.2 下一轮改进

| 改进 | 说明 |
|------|------|
| 降低误报率 | 严格 prompt: "每个发现必须含 3 项证据 + 复现步骤 + 主 Agent 验证" |
| 加证据格式 | 强制 JSON 输出: {url, method, response, screenshot_path, verdict} |
| 限定时间盒 | 60分钟硬限,超时降级 |
| review 必查 | review agent 默认必查每条发现 |

### 5.3 投入产出

- **投入**: 5 agent × 60-90 分钟 + 主 Agent 30 分钟审
- **产出**: 1 个真 P0 (退课), 2 个降级 P1, 5 个误报 (省下垃圾工作时间)
- **净收益**: 1 个真 P0 / 4.5 小时,性价比高
